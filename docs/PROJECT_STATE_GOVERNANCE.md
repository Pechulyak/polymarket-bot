# PROJECT_STATE GOVERNANCE

Версия: v1.1  
Дата: 2026-04-19

---

## 1. НАЗНАЧЕНИЕ ДОКУМЕНТА

`PROJECT_STATE.md` — это источник истины о **текущем состоянии системы**.

Документ отвечает на вопрос:

> Что есть в системе и работает ли оно?

Документ НЕ отвечает на вопросы:

> Сколько система заработала?
> Какие сейчас метрики по китам?
> Какой текущий баланс?

Источник операционных метрик:
- materialized views (paper_portfolio_state, whale_pnl_summary, paper_simulation_pnl)
- прямые SQL-запросы к БД
- специализированные dashboards (Grafana)

---

## 2. ПРИНЦИПЫ ВЕДЕНИЯ

### 2.1 Только текущее состояние
В документе хранится:
- только актуальный статус
- только последнее изменение
- только активная проблема (если есть)

История изменений не хранится.

---

### 2.2 Один объект — одна запись
Для каждой сущности:
- одна запись
- один статус
- одна актуальная проблема (если есть)

---

### 2.3 Без дублирования источников истины

| Тип данных | Где хранится |
|-----------|-------------|
| Схема БД | `init_db.sql` |
| Метрики и количества | materialized views, SQL |
| История задач | `TASK_BOARD.md` |
| Подробные изменения | git / changelog |

---

## 3. СТРУКТУРА PROJECT_STATE

Фиксированная структура:

- РЕЖИМ ПРОЕКТА
- БАЗА ДАННЫХ
- КОНТЕЙНЕРЫ
- ОСНОВНЫЕ PYTHON-ФАЙЛЫ
- PIPELINE
- БЕЗОПАСНОСТЬ
- АКТИВНЫЕ БЛОКЕРЫ
- ДОКУМЕНТАЦИЯ

Изменение структуры запрещено без отдельной задачи.

---

## 4. ФОРМАТ ЗАПИСИ

Каждая сущность оформляется строго по шаблону:

```md
- <название сущности>
  status: <статус>
  updated: YYYY-MM-DD
  task: <TASK_ID>
  note: <2-3 слова>
  issue: <кратко, если есть>
```

**Пример (с ошибкой)**
```md
- whale_trades
  status: IN_WORK
  updated: 2026-03-23
  task: TRD-420
  note: staging redesign
  issue: ingestion incomplete
```

**Пример (после фикса)**
```md
- whale_trades
  status: OK
  updated: 2026-03-23
  task: TRD-420
  note: ingestion fixed
```

---

## 5. ДОПУСТИМЫЕ СТАТУСЫ

Использовать только:

- OK
- IN_WORK
- FROZEN
- BLOCKED
- PARTIAL
- DEPRECATED
- ACTIVE

Добавление новых статусов запрещено.

---

## 6. ЦЕЛЬ ДОКУМЕНТА

PROJECT_STATE отвечает на вопрос:
  "Что есть в системе и работает ли оно?"

PROJECT_STATE НЕ отвечает на вопросы:
  "Сколько система заработала?"
  "Какие сейчас метрики по китам?"
  "Какой текущий баланс?"

Источник операционных метрик:
- materialized views (paper_portfolio_state, whale_pnl_summary,
  paper_simulation_pnl)
- прямые SQL-запросы к БД
- специализированные dashboards (Grafana)

---

## 7. ЗАПРЕТЫ

В PROJECT_STATE.md запрещено включать:

- логи
- SQL-запросы
- код
- примеры данных
- длинные описания задач
- root cause analysis
- audit-отчёты
- исторические изменения
- старые ошибки
- **любые операционные метрики**: конкретные суммы bankroll, ROI,
  realized_pnl, счётчики (hot_count, warm_count), количество 
  китов, количество трейдов
- **wallet-адреса**: конкретные whale-адреса в note/issue полях
- **временные отметки внутри note**: "Ручной запуск 09:33 UTC",
  "2189 китов обновлено"
- **даты конкретных запусков**: "reset: 2026-04-05 19:23:55 UTC"

---

## 8. РАБОТА С ОШИБКАМИ

Если ошибка есть:

- указывается в поле issue

Если ошибка исправлена:

- поле issue удаляется
- запись обновляется

Хранить историю ошибок запрещено.

---

## 9. ОБНОВЛЕНИЕ ДОКУМЕНТА

Обновление PROJECT_STATE.md выполняется:

- при изменении состояния системы
- после завершения задач
- при появлении/исправлении ошибок

Каждое обновление должно:

- менять только релевантные блоки
- не добавлять историю
- не дублировать информацию из других источников

---

## 10. КОНТРОЛЬ СОБЛЮДЕНИЯ

Нарушением считается:

- добавление логов или SQL
- добблирование данных из materialized views
- добавление исторических описаний
- использование неразрешённых статусов
- отсутствие task или updated
- операционные метрики (bankroll, ROI, realized_pnl, счётчики)
- wallet-адреса в note/issue полях

При обнаружении:

- требуется отдельная задача на cleanup

---

## 11. ПРИНЦИП МИНИМАЛИЗМА

Каждая запись должна быть:

- короткой
- однозначной
- читаемой за секунды

Если запись требует больше 1–2 строк объяснения — она не должна быть в PROJECT_STATE.
