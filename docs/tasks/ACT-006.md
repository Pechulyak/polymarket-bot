# ACT-006 — account_daily_position_ledger (mark-to-market витрина позиций)

**Статус:** DESIGN (реализация не начата, дизайн зафиксирован по итогам
обсуждения с оператором + второе мнение Fable)
**Эпик:** ACT
**Связь:** заменяет `v_position_lifecycle` (ACT-005) — та view признана
архитектурно неверной, см. §1. Порождает связанные задачи ACT-007, ACT-008,
ACT-009 (см. §8).

---

## 1. Симптом / почему ACT-005 неверна

`v_position_lifecycle` считала P&L позиции как `sell_usdc + redeem_usdc − buy_usdc`
(только реализованные денежные потоки), без учёта стоимости ещё не проданного
остатка. Для любой не до конца закрытой позиции это даёт псевдо-убыток на всю
сумму, потраченную на непроданный остаток.

Обнаружено на реальных данных, рынок «Nigel Farage win 60–70%…» (Justfuuun):
куплено на $53.08, продано на $12.71, остаток 109.7 шер по текущей цене $0.34
($37.30). Дашборд показывал `net_usd = -24.09`. Корректный mark-to-market
расчёт (продано + остаток по текущей цене − куплено = 12.71+37.30−53.08 =
**−$3.07**, плюс reward/fees) → итог **≈ +$12.5**. Знак результата был
противоположный. Ручной расчёт оператора сверен и подтверждён.

Вывод: баг систематический, а не точечный — затрагивает любую OPEN-позицию.
Использовать `v_position_lifecycle` для выводов об убыточности контура нельзя.

---

## 2. Цель

Единая витрина «экономика позиции по дням»: строки на
`(account, condition_id, asset, activity_date)`, с корректным mark-to-market
учётом (остаток оценивается по текущей цене, а не игнорируется).

---

## 3. Грейн: разреженные дни, не сплошной календарь

**Важное уточнение оператора:** строка по дате включается в витрину, только
если хотя бы одно из `buy_usdc`, `sell_usdc`, `reward_usd` за этот день ≠ 0.
Если все три — 0 (типичный случай copy-trading: купил и держит, наград нет),
строка НЕ создаётся. Это не полный `generate_series`-спайн по календарю —
набор дат = объединение (дни со сделками) ∪ (дни с reward>0, для farm-рынков).

`opening_balance`/`closing_balance` при этом всё равно считаются как running
cumulative(buy−sell) НА МОМЕНТ показанной даты — соседние показанные строки
могут быть разделены пропущенными "пустыми" днями, баланс просто переносится
через них без отдельной строки.

Витрина прекращает генерировать строки по позиции после даты закрытия
(см. §6) — не тянется до `today` для уже закрытых позиций.

## 5. Механизм расчёта

### 5.1 Материализация, не pure-SQL view

Running weighted-average-cost — stateful-расчёт. Решение (по Fable + прецедент
`farming_snapshot.py`): **Python-скрипт, материализующий таблицу**
`account_daily_position_ledger`, не recursive CTE в SQL view. Причины:
корректность stateful-логики дешевле проверить юнит-тестами на синтетических
последовательностях сделок, чем отладить в SQL. Тонкий SQL `view` поверх
таблицы — презентационный слой для Grafana, не более.

### 5.2 Full rebuild, не инкрементальный upsert

Источник (`account_activity`) append-only, объём мал. Полная пересборка с нуля
при каждом запуске — секунды, убирает целый класс багов дрейфа
инкрементального состояния.

### 5.3 Эпохи позиции (cost basis reset)

Если позиция ушла в 0 (buy≈sell) и была куплена заново — cost basis
сбрасывается, "эпоха" начинается заново. Обязательно к реализации.

### 5.4 Комиссия: universal rate, справочная колонка, НЕ доп. вычет

Проверено напрямую на живом CLOB API (`GET /markets/{condition_id}`, 4 рынка
разных категорий — 2 спорт, 2 политика): `maker_base_fee` = `taker_base_fee`
= 1000 везде. **Гипотеза "% зависит от категории рынка" не подтвердилась** —
ставка платформо-универсальная. Используем существующую калиброванную
формулу TRD-448: `fee = 0.03 × max(price, 1−price) × notional`.

**Важно:** на реальных данных Farage обнаружено, что `usdc_size` в
`account_activity` для части сделок УЖЕ включает комиссию (не равен
`price×size`), для части — нет. Поэтому `buy_fee`/`sell_fee` — это
**расшифровка**, сколько из потраченного/полученного могла составлять
комиссия, а НЕ отдельное вычитаемое значение поверх `usdc_size` (иначе
задвоим). Для гипотетического закрытия остатка (см. §5.5 mark-to-market)
комиссия ВЫЧИТАЕТСЯ из гипотетической выручки — там она реальна, а не
дублирует уже состоявшийся cash-flow.

Maker/taker по конкретной сделке подтверждаемо через on-chain лог
`OrderFilled` (CTF Exchange, Polygon) — проверено вручную на примере
US-Iran (см. §8, ACT-009): наш адрес был `maker` в обоих филлах. Это рабочий,
но дорогой (RPC-запрос на сделку) способ проверки; массовое сопоставление —
через CLOB `/trades` (ACT-008), on-chain — точечная верификация.

### 5.5 Mark-price: иерархия источников, доступна только пока OPEN

Приоритет (от точного к менее точному):
1. `account_positions_snapshot.cur_price` — с биржи, но только последние ~5
   дней и только пока API считает позицию активной.
2. `farming_daily_snapshot.mid` — только 13 farm-рынков, шире по датам.
3. CLOB `/prices-history` backfill (ACT-007, ЕЩЁ НЕ реализовано) — единственный
   источник для старых дат вне farm и вне 5-дневного окна снапшотов.

**Жёсткое правило:** `NULL mark ⇒ NULL MTM`, никогда `COALESCE(mark_price, 0)`
или `COALESCE(mark_price, avg_cost)`.

**P&L оператора (принято как основная формула):** реализованный P&L по
факту закрытых сделок + для ещё не закрытой части — гипотетический выход
ПРЯМО СЕЙЧАС по текущей цене как taker (mark_price минус расчётная taker-fee
на гипотетический выход). Пример на Farage — см. §7.

### 5.6 Границы дня

Единая точка определения "дня" (UTC) — и для агрегации сделок, и для
бакетирования `/prices-history`.

## 6. Закрытие позиции: разные механизмы для farm и copy-trading

**Основной признак (везде): `buy_size ≈ sell_size`** (dust-порог, например
1e-4 шер) → `status = CLOSED_TRADED`.

**Дополнительно для copy-trading позиций** (не фарминг — фарм обычно
закрывается через сделки раньше резолюции рынка): если позиция НЕ закрылась
по сделкам, проверяем `market_resolutions.is_closed`:
- рынок разрешён, наш `asset` НЕ выигравший (`winner_outcome`/`winner_index`
  из `tokens`) → `status = CLOSED_RESOLVED_LOSS`, `mark_price = 0`,
  принудительная терминальная дата, **независимо от наличия REDEEM-события**
  в `account_activity` (REDEEM на 0 USDC систематически теряется при дедупе,
  см. ACT-009 — полагаться на его наличие нельзя).
- рынок разрешён, наш `asset` выигравший, но REDEEM не найден →
  `status = WON_UNCLAIMED` (отдельный статус, не путать с OPEN — деньги
  реально ждут получения, не "хотим не хотим держим").
- рынок не разрешён → `status = OPEN`.

`market_resolutions` — таблица уже существует и наполняется
(`scripts/fetch_market_resolutions.py`, 71k строк, обновляется под
`whale_trade_roundtrips`-универсум), покрывает 70 из 81 наших рынков на
момент проверки (2026-07-19). Пример: рынок US x Iran (см. §8) через эту
таблицу подтверждён как `is_closed=false` — то есть ещё не разрешён, и
остаток нужно оценивать по текущей цене, а не списывать в 0.

## 7. Пример: Nigel Farage (Justfuuun), полный расчёт

| дата | откр. | buy | avg_cost | sell | sell_px | закр. | mark | reward | fee_farm |
|---|---|---|---|---|---|---|---|---|---|
| 07-15 | 0 | 100 | 0.3746 | 0 | – | 100.00000 | 0.385 | 1.995957 | 0.587412 |
| 07-16 | 100.00000 | 0 | 0.3746 | 0 | – | 100.00000 | 0.385 | 1.944218 | 0 |
| 07-17 | 100.00000 | 48.8 | 0.3567 | 34.09375 | 0.320 | 114.70625 | 0.335 | 0.969466 | 0 |
| 07-18 | 114.70625 | 0 | 0.3567 | 5 | 0.360 | 109.70625 | 0.34 (farm) / 0.33 (снапшот) | 11.952982 | 0 |
| 07-19 | 109.70625 | 0 | 0.3567 | 0 | – | 109.70625 | 0.34 (снапшот) | ещё нет | – |

Статус на 07-19: **OPEN** (buy=148.8 ≠ sell=39.09375). Обнаруженное
расхождение mark на 07-18 (farm mid 0.34 vs снапшот-с-биржи 0.33) —
два разных момента снятия цены в течение дня, оба легитимны; нужно
зафиксировать приоритет источника в реализации (§5.5).

Итог по операторской формуле: продано (факт, $12.71) + гипотетический
taker-выход остатка 109.70625@0.34 за вычетом расчётной комиссии ($36.56
net) − куплено (факт, $53.08) = **−$3.81** трейдинг + reward $16.86 − fee
farm $0.59 = **≈ +$12.47** итого. Против **−$24.09** в старом дашборде.

## 8. Смежные находки → отдельные задачи

- **ACT-007** (backfill): CLOB `/prices-history` по всем торгованным токенам
  → `mark_price`/`mark_source` для дат вне покрытия снапшотов/farm.
- **ACT-008** (backfill): CLOB `/trades` maker/taker matching → более точные
  `buy_fee`/`sell_fee`/`fee_source` (ниже приоритет, вклад в P&L меньше MTM).
- **ACT-009 (DONE)**: `account_activity` теряла реальные сделки при коллизии
  дедуп-ключа `(tx_hash, condition_id, event_type, side, size, price)` — если
  в одной транзакции два РАЗНЫХ ордера (разный `orderHash`) дают одинаковый
  размер/цену, второй молча не писался. Подтверждено on-chain (Polygon RPC,
  `eth_getTransactionReceipt`, событие `OrderFilled` CTF Exchange).

  Полное расследование: проверены все 274 уникальных tx_hash по TRADE-событиям
  обоих аккаунтов (не только Justfuuun-CSV). Найдено 52 расхождения
  (on-chain-событий больше, чем строк в БД), из них при перепроверке —
  **только 3 настоящие коллизии** (идентичный размер/цена у двух РАЗНЫХ
  ордеров в одной роли — наш индекс их гарантированно схлопывает):
  - `0x7b0209b7...` (US x Iran, Justfuuun, MAKER 200/$148 ×2) — бэкфиллено.
  - `0x3aae7dd4...` (Justfuuun, TAKER 50.0/44.5 ×2) — **не бэкфиллено**, см. ниже.
  - `0xeaf3b573...` (Justfuuun, TAKER 50.0/32.0 ×2) — **не бэкфиллено**, см. ниже.

  Остальные 49 расхождений — **не баг**: один пользовательский трейд на
  Polymarket нередко исполняется как 2 on-chain события (наша нога taker +
  встречная maker в одной транзакции, суммы согласуются между собой),
  Data-API корректно сворачивает это в одну запись активности. Первоначальный
  подсчёт "on-chain событий больше, чем строк в БД" ошибочно считал оба
  события как обязанные быть отдельными строками.

  Фикс (применён на S1, confirmed): `migrations/migration_act009_activity_fill_seq.sql`
  — добавлена колонка `fill_seq` (дискриминатор, т.к. Data-API не отдаёт
  order-level ID — `raw_json` для задублированных филлов байт-идентичен),
  unique-индексы пересозданы с её учётом. Бэкфилл: одна строка для US x Iran
  (`SELL 200/$148, fill_seq=1`). После бэкфилла и пересборки леджера —
  позиция корректно закрыта (`buy=sell` с точностью до пыли), P&L = **−$111.39**
  (было −259.39 до фикса, а до находки ACT-006 дашборд показывал −24.09).

  Побочный фикс в `build_account_daily_ledger.py`: dust-epsilon поднят с
  `1e-4` до `0.05` — на позиции с ~98 сделками накопленная погрешность
  float-суммирования (0.0015) превышала старый порог, из-за чего закрытая
  позиция ошибочно оставалась `OPEN`. Проверено по всему датасету — других
  ложно-открытых dust-позиций не осталось (минимальный размер ордера на
  Polymarket ~5 шер, порог 0.05 безопасен).

  **Новая, более серьёзная находка — ACT-010** (см. ниже): 2 оставшихся
  случая (`0x3aae7dd4...`, `0xeaf3b573...`) при декодировании оказались НЕ
  дублями существующих позиций — задублированные филлы относятся к asset ID,
  которого **вообще нет нигде** в `account_activity` (ни у одного аккаунта,
  ни по одному рынку), хотя обе транзакции — внутри окна трекинга (после
  06-19). Это значит Data-API не сообщил о целой активности вовсе, а не
  просто "потерял одну строку из двух одинаковых". Не бэкфиллено — не хватает
  title/slug/условия для этих assets (нужен доп. запрос к Gamma API), и
  неясно, сколько ещё всего таких непрослеженных активностей могло быть за
  всё время (эта проверка покрывала только TRADE-транзакции, которые УЖЕ
  что-то записали в БД — транзакции, где Data-API не сообщил вообще ничего
  ни по одному активу, этой проверкой не находятся в принципе).

- **ACT-010 (новая, BACKLOG)**: возможные полностью непрослеженные позиции —
  Data-API `/activity` не сообщает часть taker-сделок, когда в той же
  транзакции есть другая (maker) активность. Обнаружено 2 конкретных
  экземпляра (см. выше), оба — TAKER-филлы Justfuuun по assets, которых нет
  в БД совсем. Требует: (1) резолвинг title/condition_id этих 2 assets через
  Gamma API и решение — восстанавливать ли задним числом; (2) более широкая
  проверка — сравнить полную on-chain историю ERC-1155 Transfer-событий на
  CTF-контракте для обоих proxy_wallet с тем, что есть в account_activity, а
  не только внутри уже частично записанных транзакций (методология этой
  сессии структурно не может найти транзакции, полностью пропущенные Data-API
  целиком).
