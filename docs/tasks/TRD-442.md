# TRD-442 — DB-trigger закрытия paper-позиций на SELL

**Эпик:** ARCH-PAPER-SELL-TRIGGER
**Статус:** IN_PROGRESS
**Создан:** 2026-05-15
**Приоритет:** P0 для paper trading
**Параллельный эпик:** ARCH-3BC-PIPELINE (не блокирует, другой исполнитель)

## Контекст

Paper-киты копируют BUY-сделки через существующий trigger `trigger_copy_whale_trade`.
SELL-сделки сейчас не обрабатываются — paper-позиция остаётся OPEN до резолюции рынка
и закрывается через `run_settlement.sh` cron по цене 1.0/0.0.

Это приводит к искажению paper-метрик: если кит вышел при цене 0.75, а рынок
резолвится в 1.0 — paper-позиция закрывается по неправильной цене, реальная цена
выхода теряется.

## Цель

Добавить DB-trigger `trigger_close_paper_on_sell`, который синхронно в commit
`INSERT INTO whale_trades WHERE side='sell'` закрывает соответствующую OPEN
paper-позицию по реальной цене SELL.

## Архитектурные решения (зафиксировано Master)

- Механизм: DB-trigger, не cron, не Python.
- Срабатывание: AFTER INSERT ON whale_trades WHERE side='sell'.
- Закрывает только paper-позиции (whales.copy_status='paper').
- Trigger трогает только paper_trades (симметрия с trigger_copy_whale_trade).
- НЕ трогает whale_trade_roundtrips в первой версии (см. Q1).
- run_settlement.sh продолжает работать как страховка.

## Открытые вопросы (предварительные решения Dev-чата)

### Q1 — Trigger на roundtrip в этой же транзакции?
**Решение: НЕТ.** Trigger трогает только paper_trades. Roundtrip-закрытие
остаётся за cron 3C из параллельного эпика ARCH-3BC-PIPELINE. Симметрия
теряется на ≤1 час, это приемлемо.
Обоснование: принцип "один trigger — одна ответственность"; избегаем конфликта
с параллельным эпиком.

### Q2 — Расчёт net_pnl_usd в trigger
**Решение: использовать существующую формулу из settle_resolved_positions().**
Без адаптации. Конкретный набор fee-полей фиксируется в TRD-442-C после
чтения paper_position_settlement.py и SQL settlement-функции.

### Q3 — Что делать с уже OPEN paper-позициями, чьи SELL прошли в прошлом?
**Решение: (a) forward-only.** Trigger работает только для новых SELL.
Backfill исторических SELL — отдельная задача после стабилизации, не блокирует
ре-активацию.

⚠️ Эти решения подлежат финальному подтверждению Master до старта TRD-442-C.
