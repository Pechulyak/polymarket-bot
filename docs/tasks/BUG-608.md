# BUG-608 — Закрытие позиций roundtrip_builder не выполняется в проде

**Статус:** TODO  
**Эпик:** BUG  
**Тег:** feature:roundtrip-close  
**Severity:** HIGH  
**Обнаружен:** 2026-05-05 (debug-чат, в рамках TRD-403)  
**Связь:** TRD-403 (DONE — верификация, выявившая баг)

---

## 1. Симптом

В `whale_trade_roundtrips` за всю историю таблицы (47 087 записей):

- `close_type = 'SELL'` — **0 записей**
- `matching_method = 'FLIP'` — **0 записей**
- `close_side = 'sell'` — **0 записей**
- `matching_confidence = 'MEDIUM'` — **0 записей**

Все 39 348 закрытых roundtrips закрыты исключительно через резолюцию рынка
(`SETTLEMENT_WIN` / `SETTLEMENT_LOSS`).

---

## 2. Root Cause

Контейнер `polymarket_roundtrip_builder` в `docker-compose.yml` запускается командой:
python -m src.strategy.roundtrip_builder

Без аргументов. В `main()` файла `src/strategy/roundtrip_builder.py` это
default-ветка argparse → `builder.run()` (BUY → OPEN pipeline, ARC-502-A).

Sell-pipeline реализован как `builder.run_close_positions()` и вызывается только
при флаге `--close`. Этот флаг **нигде в проде не передаётся** — ни в
docker-compose, ни в crontab.

Функция `_close_roundtrips()` в коде есть, написана корректно, имеет fuzzy
matching для short-sell случаев — но никогда не выполнялась в production.

## 4. Архитектура roundtrip_builder

Файл: `src/strategy/roundtrip_builder.py`

Три entry point'а через argparse в `main()`:

| Флаг | Метод | Что делает | Где сейчас запускается |
|------|-------|-----------|------------------------|
| (default) | `builder.run()` | BUY → создаёт OPEN roundtrips | docker-compose, контейнер `polymarket_roundtrip_builder`, цикл `sleep 7200` |
| `--close` | `builder.run_close_positions()` | SELL → закрывает OPEN roundtrips, ставит `close_type='SELL'`, `matching_method='FLIP'` | **НИГДЕ** |
| `--settle` | `builder.settle_roundtrips_via_gamma()` | резолюция рынка → `SETTLEMENT_WIN/LOSS` | предположительно через `run_settlement.sh`, cron `0 */2 * * *` |

Внутренние функции sell-pipeline:

- `_fetch_and_group_sell_trades()` — `WHERE wt.side = 'sell'` в `whale_trades`,
  агрегирует по `(wallet, market, outcome)`.
- `_close_roundtrips()` — для каждого sell-агрегата находит OPEN roundtrip
  (точное совпадение по `position_key`, иначе fuzzy по `wallet+market`),
  считает `gross_pnl = (close_price - open_price) * close_size`, обновляет
  roundtrip с `close_type='SELL'`, `matching_method='FLIP'`,
  `matching_confidence='MEDIUM'`, `pnl_status='CONFIRMED'`.
- `_update_whales_pnl()` — после закрытия пересчитывает агрегаты в `whales`
  (`total_pnl_usd`, `win_count`, `loss_count`, `win_rate_confirmed`).

CHECK constraint `close_type` в БД: `('SELL', 'SETTLEMENT_WIN', 'SETTLEMENT_LOSS', 'FLIP')`.
Значение `'FLIP'` существует в constraint, но в коде нигде не присваивается
(присваивается `'SELL'` с `matching_method='FLIP'`). Зафиксировано как побочная
находка, некритично.

## 5. Последствия для системы

1. **Все sell-операции китов теряются** на уровне реконструкции позиций. Кит,
   который купил YES @ 0.30, продал @ 0.80 (профит), а потом рынок зарезолвился
   в LOSS — у нас записан как `SETTLEMENT_LOSS` на оригинальную позицию.
   Реальный профит кита потерян.
2. **WR и PnL китов в `whales` искажены** — считаются только по позициям,
   дошедшим до резолюции. Активные торговцы выглядят как hold-to-resolution.
3. **Classification (paper / tracked / excluded) основана на искажённой картине.**
   Возможна ре-оценка ряда китов после фикса (включая 0x32ed, недавно переведённый
   в tracked).
4. **`paper_simulation_pnl` транслирует искажение в копию.** Мы копируем
   «фантомного» кита, который только покупает и держит до конца. Реальные
   торговые решения кита (выход с прибылью, стоп-лосс) не воспроизводятся.
5. **Объём накопленного искажения большой** — 47 087 закрытых roundtrips
   должны были иметь sell-варианты, но не имеют.
6. **`pipeline_monitor` (cron `*/30`) не отловил.** Технический долг — нужен
   health-check на класс «sell trades в whale_trades > 0, но новых SELL-roundtrips
   за тот же период = 0».

## 6. Варианты решения (на согласование с STRATEGY)

### Вариант 1 — Минимальный фикс в docker-compose (рекомендуется debug-чатом)

```yaml
command: >
  sh -c "while true; do
    python -m src.strategy.roundtrip_builder;
    python -m src.strategy.roundtrip_builder --close;
    sleep 7200;
  done"
```

- **Плюсы:** одна правка, минимальный blast radius, тот же контейнер, тот же sleep.
- **Минусы:** последовательность важна (сначала `run()` создаёт OPEN, потом
  `--close` закрывает их через sell). Если упадёт первая команда — вторая не
  выполнится; следующий цикл попробует снова.

### Вариант 2 — Отдельный контейнер для close

Поднять `roundtrip_builder_close` с `--close` и собственным циклом.

- **Плюсы:** изоляция ошибок, отдельные логи, можно дать другую частоту.
- **Минусы:** ещё один контейнер в docker-compose, больше overhead.

### Вариант 3 — Отдельный cron вместо контейнера

По аналогии с `run_settlement.sh`. Сделать `run_close_positions.sh`, добавить
в crontab каждые 2 часа.

- **Плюсы:** консистентность с settlement-path.
- **Минусы:** разводит логику по двум разным механизмам (docker-compose loop +
  cron) для близких задач.

## 7. Варианты обработки истории (на согласование со STRATEGY)

После включения `--close` следующий запуск обработает все sell-трейды за всё
время (47 087 потенциальных кандидатов). Это:

- Создаст соответствующие SELL-roundtrips или модифицирует существующие OPEN.
- Перепишет агрегаты в `whales` через `_update_whales_pnl()` — изменятся
  `total_pnl_usd`, `win_count`, `loss_count`, `win_rate_confirmed`.
- Может изменить classification у некоторых китов.
- `paper_simulation_pnl` view будет отдавать другие цифры на тех же
  `paper_trades` (присоединение к `whale_trade_roundtrips`).

### Вариант A — Полный rollout (рекомендуется debug-чатом)

Запустить один раз на всём датасете, принять единовременную перетряску метрик.
Историю whale_pnl до этой точки считать «эпохой 1», после — «эпохой 2».

### Вариант B — Только свежие данные

Запустить только начиная с какой-то даты, оставить историческое окно как есть.
Пограничные roundtrips (OPEN в «старой эпохе», SELL в «новой») — особый случай.

### Вариант C — Полный rebuild closed-roundtrips

`--rebuild` всех closed-roundtrips и пересборка с нуля. Самое чистое, но самое
травматичное.

## 8. План фикса (предварительный, ≤5 шагов)

1. Backup `whale_trade_roundtrips` (snapshot для отката).
2. Правка `docker-compose.yml` (Вариант 1 после согласования).
3. **Тестовый прогон только на 0x479e** (17 sell-трейдов, контролируемый scope).
4. Верификация на 0x479e: появились ли SELL-roundtrips, изменились ли его
   метрики в `whales`, что отдаёт `paper_simulation_pnl` на его paper_trades.
5. Stop-point — решение о broad rollout эскалируется в мастер-чат.
