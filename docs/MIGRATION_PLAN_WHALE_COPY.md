# План миграции: оптимизация whale_copy-контура (DRAFT)

> Статус: **DRAFT** — черновик, собран 2026-07-21. Ничего ещё не удалено/не изменено.
> Цель: перед переездом на новый сервер убрать «громоздкую инфраструктуру с нулевым
> выхлопом» вокруг whale_copy (приоритет снижен), оставив компактный контур из
> ~15 реально копируемых китов + farming.

---

## 1. Диагноз текущего состояния (по живой БД/логам, 2026-07-21)

### 1.1. Реальный полезный контур — 15 китов
Копи-пайплайн использует только китов с активным `copy_status`:

| copy_status | кошельков | whale_trades | roundtrips |
|---|---|---|---|
| excluded | 106 | 161 048 (~50 MB) | 97 779 CLOSED |
| paper | 2 | 133 233 (~48 MB) | 2 517 |
| tracked | 12 | 103 356 (~38 MB) | 12 345 |
| none (discovery-масса) | 21 981 | 74 034 (~24 MB) | 55 954 |
| live | 1 | 6 471 | 4 642 |

Полезны для копирования: **15** (12 tracked + 2 paper + 1 live). Остальное — накладные
расходы discovery.

### 1.2. Объёмы и рост
- `whale_trades` — 612 MB / 478K строк
- `whale_trade_roundtrips` — 214 MB / 184K строк
- `whales` — 85 MB / 26 956 строк
- Рост whale_trades экспоненциальный: май 62K → июнь 130K → **июль 246K**.
- Discovery добавляет ~3–8K новых китов в месяц; **6 252** `none`-кита не дали
  ни одного roundtrip (чистый шум).

### 1.3. Развенчание мифов
- **roundtrip_builder НЕ считает 2 часа** — по логам контейнера отрабатывает
  **70–90 сек**; «2 часа» = `sleep 7200` между прогонами. Трогать не нужно.
- **Настоящие «2 часа» — `category_backfill`**: `SELECT DISTINCT market_id
  WHERE market_category IS NULL` → на каждый рынок HTTP `clob.polymarket.com/markets/{id}`,
  большинство которых уже **404 (мёртвые рынки)**. Мёртвые рынки никогда не
  резолвятся, остаются `NULL` и **перевыбираются каждый прогон** → бесконечный
  churn (cumulative обработано 1.18M, обновлено 103K, предельный выхлоп ~0).
  `flock -n` не даёт уложиться в 2ч-интервал — процесс идёт практически всегда.

### 1.4. Побочные находки
- **account_activity_fetch** — не сломан по сути (данные свежие, `ingested_at`
  сегодня 10:29). Но ночной cron 04:10 падает на ветке REDEEM
  (`ON CONFLICT (tx_hash, condition_id, event_type, size, fill_seq)` — нет такого
  constraint). Основная активность пишется, redeem-события — нет. Узкий баг, не блокер.
- **excluded-киты держат данные вечно** — retention чистит только по возрасту
  (30д), не по статусу. 106 excluded = 161K сделок + 98K roundtrips заморожены навсегда.

## 2. Решения по каждому направлению

### 2.1. category_backfill — чинить (это аналитика), иначе отключить
Нужен для аналитики, но грузит сервер впустую. Варианты по возрастанию радикальности:
1. **Negative-cache** (минимальный фикс): помечать нерезолвящиеся (404) рынки как
   `category='unknown'`, чтобы не перевыбирать их каждый прогон → churn сходится.
2. **Bulk через gamma-api** вместо поштучного clob: `gamma-api/tags` + `/events?tag_id=`
   даёт категории пачкой по открытым событиям (без auth). Для мёртвых рынков всё равно
   нужен negative-cache.
3. **Категория кита вместо категории рынка** (предложение оператора): для leaderboard
   мы уже классифицируем категорию кита. Если market-level категория используется только
   в аналитике — перейти на whale-level и **выпилить market-backfill целиком**.
4. **Отключить** (fallback): если 1–3 не взлетают — остановить cron, категорию рынка
   не собирать. Приемлемо, т.к. это исключительно аналитика.

Действие сейчас: **анализ**, не чинить. Определить, где реально потребляется
`market_category` (аналитика/фильтры), — от этого выбор 1/2/3/4.

### 2.2. Массовый excluded + удаление данных
Правило: перевести бесперспективных китов в `excluded`, **удалить их whale_trades и
roundtrips**, в `whales` оставить строку + `whale_comment` (причина исключения).
Важно: сегодня `excluded` только останавливает построение новых roundtrips
(`copy_status != 'excluded'` в builder), но **не удаляет** накопленное — нужен
отдельный шаг удаления. Это money-adjacent (DELETE) → через confirmed-гейт.
Отбор «кого гасить» — отдельное аналитическое суждение по каждому киту.

### 2.3. Ужесточить retention
Добавить к возрастному retention удаление по статусу: excluded-киты и `none`-киты
без единого roundtrip (6 252 шт.). Оценка сложности/blast radius — в §4.

### 2.4. Отключить discovery
Делаем. Тracked-poll (5 мин по `qualification_status IN ('qualified','ranked','tracked')`)
— отдельный механизм, копирование от него не зависит. Останавливает рост таблиц у корня.

### Шаг 2 — category_backfill (MIG-002) — ✅ DONE
**Итог:** истинная первопричина churn — баг ingestion (малформенные len-64 market_id,
вынесен в TRD-451), не дизайн backfill. Фикс: convergence-снапшот + фильтр length=66 +
удаление Step 2 API. Прогон конечный (2м11с vs 9 суток), крон возвращён (раз в сутки).
Уровень B (структурный 404/age-guard/gamma/DDL) отклонён вторым мнением как неверная цель.
Детали ниже — история анализа.
**Анализ (подтверждён кодом + второе мнение Opus):**
- Крон не завершается никогда (висел 9 суток), вечно чурнит мёртвые (404) рынки; из-за
  этого roundtrip-фаза (Step 1 JOIN) не запускалась НИ РАЗУ → 51.5K NULL-категорий.
- Модель «свежие=NULL, мёртвые=unknown» ОПРОВЕРГНУТА: детектор пишет None
  (whale_detector.py:1271), repo дефолтит None→'unknown' (whale_trades_repo.py:153) —
  КАЖДАЯ сделка входит 'unknown' (флаг «ждёт backfill»), NULL в whale_trades нет.
  Из 1876 'unknown' 482 свежие ≤3д, 11 >30д → терминализация «по None» замуровала бы живые.
- Альтернатива «категория кита» — не подходит (иная семантика + пусто). Gamma не нужен.
- Потребители market_category — только аналитика (weekly_whale_analysis, whale_status, монитор).

**Сделано:**
- ✅ **Часть 1:** разовый SQL-JOIN roundtrips←whale_trades (real-only) — 49 754 строки
  категоризированы без API; остаток 1825 NULL = 'unknown'-рынки (orphan=0). Идемпотентно.
- ✅ **Интерим:** крон category_backfill приостановлен (crontab-протокол, бэкап+diff+confirmed),
  9-суточный процесс погашен, churn остановлен. Категоризация свежих на паузе.

**Pending — редизайн (полный фикс churn):**
- третий статус (нельзя переиспользовать 'unknown' как терминал);
- структурное распознавание HTTP 404 в клиенте (сейчас всё → None);
- возрастной guard (≥7–14д) перед терминализацией; исключить терминальные из re-select;
- удалить Step 2 (API, not_in_wt=0 структурно); снять дублирующий loop в детекторе (DEPRECATED);
- пересмотреть pipeline_monitor.check_market_category_null_pct при выборе NULL-варианта.
- Порядок: сначала интерим (done) → потом редизайн (проект → второе мнение → код → confirmed).

### Шаг 3 — Ужесточить retention (MIG-003) — ⏸ ОТЛОЖЕН НА МИГРАЦИЮ
**Решение оператора (2026-07-22): purge НЕ выполняется на живом S1. Он будет выполнен
на ВОССТАНОВЛЕННОЙ КОПИИ на новом сервере** (миграция всё равно идёт через dump/restore):
прод не трогаем до катофа, откат бесплатный (перелить дамп), разрушительная операция —
над копией. Дополнительно: место всё равно материализуется только через dump/restore,
обычный DELETE его ОС не возвращает.

#### Анализ выполнен (read-only) — первопричина
- Возрастной retention не уменьшает таблицы: **149 390 из 150 506 сделок >30д (99.3%)
  защищены** ссылками roundtrips (FK NO ACTION), а **у roundtrips retention нет вообще**.
- Цель чистки: excluded + none. Замер: 155 276 rt / 220 482 сделки (при правиле «кит целиком»).

#### БЛОКЕРЫ (обязательны к учёту при выполнении на копии)
- **B1 — искажение вердикта.** `update_whale_pnl_from_roundtrips()` (cron `0 */2` из
  run_settlement.sh) делает FULL RECOMPUTE из CLOSED-roundtrips. Кит с ЧАСТИЧНО оставшимися
  rt пересчитается по огрызку (пример: 1106 rt / −$55 643 → 28 rt / произвольный P&L).
  Кит с НУЛЁМ оставшихся rt не пересчитывается (выпадает из подзапроса) — вердикт цел.
  → **Правило «кит целиком или никак»** + whale-level guard: 9 excluded-китов с paper-историей
  сохранять ПОЛНОСТЬЮ (7 368 rt, +2.5% объёма), а не по 4-колоночному джойну.
- **B2 — purge самоотменяется.** `roundtrip_builder.py:104-108` фильтрует только `excluded`,
  НЕ `none` → билдер (каждые 2ч) воссоздаст **16 808 roundtrips за ≤2ч**, OPEN вырастет.
  → Нужно: удалять и сделки 'none' (без возрастного гейта) **+ правка билдера**
  `NOT IN ('excluded','none')` ДО purge.
- **B3 — `whale_pnl_summary`.** LEFT JOIN от `whales` с фильтром copy_status IN
  (paper,tracked,excluded): строки excluded НЕ исчезнут, а **обнулятся** (108 «китов с 0 P&L») —
  выглядит как настоящие данные.
- **F1 — «−0.5 ГБ» неверно** для живой БД: 564 из 953 MB — индексы; DELETE место не вернёт.
  Материализуется только dump/restore (что и происходит при миграции).

#### Прочее к учёту
- **Paper-история:** 9 203 paper_trades (89%) принадлежат 9 excluded-китам; после ресета
  bankroll (2026-04-05) — 6 822. Матвьюхи paper_* джойнят по
  (market_id, lower(wallet), **open_side**, lower(outcome)) с DISTINCT ON — сохранять whale-level.
- **Блок 3 weekly** (категорийный edge) агрегирует по ВСЕМ китам без фильтра copy_status →
  база схлопнется ~174K→~22K CLOSED, ряд недель порвётся, а отчёт идёт в AI-модель.
- **199 'none'-кандидатов** (≥30 CONFIRMED rt, из них 134 c WR≥0.55) — Блок 2 weekly читает их
  через INNER JOIN к CTE из roundtrips → после purge механизм промоушна умрёт.
  → Перед purge **материализовать компактный снапшот кандидатов** (несколько сотен строк).
- **85 roundtrips с `whale_id IS NULL`** — не ловятся фильтром по copy_status, решить явно.
- Строки `whales` для none (26 839, 85 MB) — **не удалять**: единственный след «кошелёк уже
  отвергнут», выигрыш мал и без VACUUM FULL не вернётся.

#### Runbook (для выполнения на копии)
1. Правка билдера (B2) → прогон → убедиться, что регенерации нет.
2. Снапшоты: кандидаты из none; вердиктные колонки `whales` для 108 excluded; baseline всех
   полей `paper_portfolio_state`.
3. Staging-таблицы `mig003_purge_rt` / `mig003_purge_wt` (фиксируют ТОЧНЫЕ id — предикат
   дрейфует, close_sell двигает OPEN→CLOSED ежечасно) → дамп удаляемого по ним → удаление
   батчами по ним же с отметкой (возобновляемость + аудит).
4. Порядок: roundtrips (батч 5 000, `ORDER BY id`, `lock_timeout=5s`, CLOSED → потом OPEN),
   затем whale_trades (>30д — существующей процедурой; <30д для none — отдельным DELETE
   **с обоими NOT EXISTS**, иначе FK оборвёт батч).
5. `VACUUM (ANALYZE)` обеих таблиц. **VACUUM FULL на живом S1 не делать.**
6. Приёмка: рефреш 3 матвьюх CONCURRENTLY → `paper_portfolio_state` обязан совпасть с baseline
   до цента → smoke_test. Через 2ч и 4ч: вердикт 9 китов не поплыл, rt у none не растут.
7. Окно запуска вне `:00` (settlement/builder), `:15` (close_sell/refresh), `0 */4`
   (category_backfill), `04:00` (retention).

#### Побочные находки — отдельными задачами (не смешивать с MIG-003)
- `run_daily_whale_alert.py:222,268` — checks 4/5 сравнивают `status='closed'` (нижний регистр),
  а в БД `CLOSED` → **обе проверки мертвы сегодня**.
- `whale_pnl_summary` не включает `live`-китов вообще.
- Блок 3 weekly не фильтрует `copy_status`.

## 4. Blast radius / открытые вопросы (заполнить при проработке шагов)

- **retention по статусу:** кто читает старые roundtrips excluded/none? (аналитика,
  weekly_analysis, whale_pnl_summary). Проверить FK и вьюхи перед удалением.
- **discovery off:** точная точка глушения петли в `whale_detector` (где стартует
  discovery vs `_tracked_poll_loop`); влияние на `source='POLLER'` метрики.
- **category_backfill:** список потребителей `market_category` (grep по коду/вьюхам).
- **миграция без истории (опция «чистый старт»):** переносить схему + 15 китов +
  их OPEN-позиции (~2 200 строк), историю whale_trades/roundtrips оставить дампом
  на старом сервере. Снимает 826 MB и DELETE-риск. Требует решения оператора.

## 5. Целевое состояние после миграции
Компактный контур: ~15 китов (tracked/paper/live) + targeted-poll + copy-пайплайн +
farming целиком. Без широкого discovery и без бесконечного category-backfill.
Retention по статусу+возрасту не даёт узкому контуру заилиться.
