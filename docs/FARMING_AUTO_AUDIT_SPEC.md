# Спека: авто-аудит фарминга через OpenRouter + inline-кнопки TG

> **Статус: DRAFT.** Проектирование. Кода/миграций/деплоя нет. Номера тикетов —
> предложение (сверено по TASK_BOARD на 2026-07-20: последний FARM-042).
> Money-adjacent разделы прошли второе мнение Fable (2026-07-20), замечания
> внесены в §2 / §4.3 / §5.2 / §6.1-6.2 / §8.3.

## 0. Проблема и цель

Сейчас дневной мониторинг фарминга — ручной 4-итерационный цикл: TASK PACK A
(сбор данных S1 + Gamma) → мой анализ → решение оператора → TASK PACK B (деплой
на S2). Аналитическая итерация повторяема и формализуема чек-листом (методология
15.07). Цель — вынести её в LLM: регулярный авто-аудит собирает состояние,
выдаёт в TG отчёт + ранжированные рекомендации, оператор реагирует inline-кнопками.
**Исполнение — только после нажатия оператора; LLM не исполняет ничего** (money-adjacent
всегда за человеком).

## 0.1 Топологическая вводная (определяет всю архитектуру)

Разведка 2026-07-20 (`docs/tasks/FARM-043_recon.md`) зафиксировала
разнесённость по хостам:

| Компонент | Хост | Факт |
|---|---|---|
| Postgres (snapshot/candidates/positions/activity) | **S1** | localhost:5433/polymarket |
| Snapshot-пайплайн (cron 12:35/13:35 UTC) | **S1** | наполняет таблицы аудита |
| Прецедент OpenRouter (`run_weekly_whale_analysis.py`) | **S1** | уже работает cron'ом |
| `OPENROUTER_API_KEY` | **S1** (.env) | уже есть, переиспользуем |
| TG control-bot (`executor/farming_control_bot.py`) | **S2** | монополист getUpdates |
| Демон фарминга + episode-лог (halt/pause/one_sided) | **S2** | `/opt/executor/logs/farming_daemon.log` |

Данные — на S1, TG-приём и episode-логи — на S2, SSH S1→S2 закрыт. Это
центральное ограничение: **отправлять** в TG с S1 можно (Bot API sendMessage —
исходящий HTTPS, getUpdates не нужен), а **принимать** нажатия кнопок можно
только через единственный getUpdates-поток бота на S2.

**Ключевая находка, снимающая зависимость от S2-логов:** сигнал
one_sided / both-legs уже персистится в Postgres на S1 через
`farming_daily_snapshot.legs_state`, `legs_state_log`, `hours_both`. Episode-лог
демона добавляет детали pause/reconcile, но существо (обе ноги живы vs
односторонний) аудит видит из снапшота. **Фаза 1 жизнеспособна на S1 в одиночку.**

## 1. Входные данные аудита

### 1.1 Источники (все — Postgres S1, если не указано иное)

| Источник | Таблица/файл | Что даёт аудиту | Срез |
|---|---|---|---|
| Дневной снапшот фарминга | `farming_daily_snapshot` | reward_usd, fees_usd, capital_usd, inv, mid, **legs_state / legs_state_log / hours_both** | последний `snap_date` (~5 строк) |
| Позиции (mark-to-market) | `account_positions_snapshot` | cash_pnl, realized_pnl, current_value, avg_price, cur_price, end_date | последний `snap_date` (~11 строк) |
| Скринер-кандидаты | `farming_market_candidates` | rewards_min_size, rewards_max_spread, days_to_end, reward_pool_daily_rate, thin_book, dead_book, competitor_weight, est_share; `est_daily_yield_pct`/`required_capital` — **ненадёжны, см. §1.5** | последний scan-батч по `scanned_at` (~26 строк) |
| Активность аккаунта | `account_activity` | ротации/сделки за окно | последние 24-48ч по `event_ts` |
| Gamma-параметры | Gamma API (S1 out) | pool, spread, end_date (сверка с БД, при расхождении) | по held+candidate рынкам |
| Episode-лог демона | S2 `farming_daemon.log` | halt/pause/reconcile-детали | **недоступен на S1** — см. 1.4 |

### 1.2 Объём и форма — измерено, не оценено

Сырой JSON последнего дня (`SELECT *`, 42 строки): **52 262 симв ≈ 13–17k токенов**.
Простое усечение до top-20 экономит лишь ~14% — потому что раздувают
`raw_json`/`jsonb`-колонки в `account_positions_snapshot` и `account_activity`,
а не число строк.

**Решение по форме — JSON-дайджест (не таблицы), с явным pruning:**
- форма JSON, а не markdown-таблицы: LLM парсит JSON надёжнее, и это совпадает
  с прецедентом `run_weekly_whale_analysis.py` (system-prompt + user-JSON);
- **выбросить `raw_json`/jsonb-колонки** и служебные поля (id, scan_run_id,
  ingested_at, tx_hash, raw) из дайджеста — это, а не top-N, даёт основную
  экономию; ожидаемо ~8–11k токенов после pruning;
- Decimal → float (как `DecimalEncoder` в прецеденте);
- на выход LLM закладываем ~1.5–3k токенов (per-market вердикты + summary).
- **Бюджет на вызов ≈ 12–14k токенов total** — мало, укладывается в любой
  современный контекст; выбор модели не лимитируется контекстом (см. §3).

### 1.3 Структура дайджеста (для промпта)

```
{
  "as_of": "<snap_date/scanned_at>",
  "farming_state":   [ {token,name,legs_state,hours_both,inv,mid,capital_usd,fees_usd,reward_usd,reward_trend_Nd} ],
  "positions":       [ {condition_id,title,size,avg_price,cur_price,current_value,cash_pnl,realized_pnl,end_date,redeemable} ],
  "candidates":      [ {question,category,est_daily_yield_pct,rewards_max_spread,days_to_end,reward_pool_daily_rate,required_capital,thin_book,dead_book,competitor_weight,est_share} ],
  "activity_24h":    [ {event_type,side,size,usdc_size,price,title,event_ts} ],
  "gamma_overrides": [ {condition_id,pool,spread,end_date} ]   # только при расхождении с БД
}
```
`reward_trend_Nd` — производная (reward_usd последних N дней), считается на S1 до
промпта: LLM не должен «вспоминать» тренд, ему подаём готовый ряд.

### 1.4 Episode-контекст (halt/pause/one_sided)

- Существо (one_sided vs both) — из `legs_state`/`hours_both` снапшота (S1, есть).
- Детали pause/reconcile — только в S2-логе. Формат строки (подтверждён):
  `[YYYY-MM-DD HH:MM:SS] [FARM-011/012] PAUSE on <market>: reason=<...> min_left=<..> (until=<epoch>)`.
- **Фаза 1 работает без S2-лога.** Обогащение episode-деталями — опциональный
  FARM-047 (демон персистит episode в Postgres на S2, реплика доступна на S1);
  это изменение демона на S2 = вне сессии, отложено.

### 1.5 Известные дефекты входных данных (FARM-050) — учесть при реализации

Проверено по коду `farming/tools/farm_screen.py` (коммит 2227e6f, ветка
`claude/focused-lamport-74a4d1`, на момент 2026-07-20 не в main):

| Дефект | Факт (file:line) | Следствие для аудита |
|---|---|---|
| **`est_daily_yield_pct` завышен при `min_size < 100`** | `size_for_c = max(100, min_size)` (`farm_screen.py:399`) задаёт числитель `our_daily`, а `required_capital = rewardsMinSize × mid × 2` (`:318`) — знаменатель по `min_size`; деление на `:127-129` | Числитель по 100 шер, знаменатель по `min_size` → доходность завышена. **Нельзя брать `est_daily_yield_pct` из БД как факт** для порога скрининга (чек-лист п.2) и EV-сравнения (п.4) |
| **`size_used` не персистится** | в INSERT-списке колонок его нет (`farm_screen.py:106`), хранится только в памяти/result.json | Аудит из Postgres **не видит, каким размером ноги оценён кандидат** — восстановить `max(100, min_size)` нужно на стороне S1-кода из `rewards_min_size` |

**Решение для спеки (прямое следствие принципа §2 «числа из кода»):**
S1-ассемблер (FARM-043) **пересчитывает** доходность кандидата сам —
из `rewards_min_size`, `mid` и восстановленного `size_used = max(100, rewards_min_size)`,
— а не доверяет колонке `est_daily_yield_pct`. Колонка подаётся LLM (если вообще)
только как справочная, помеченная `unreliable`. Это ровно тот класс ошибки, против
которого введён принцип: неверная доходность у кнопки «Подтвердить».

Альтернатива (вне скоупа аудита): починить `required_capital`/добавить колонку
`size_used` в `farming_market_candidates` — это DDL = money-adjacent confirmed-гейт,
отдельным тикетом. До этого аудит обязан считать сам.

## 2. Логика аудита (чек-лист 15.07 — вывод, не констатация)

LLM обязан по каждому пункту **объяснять причину с цифрами**, а не пересказывать
состояние. System-prompt закрепляет методологию; выход — структурированный JSON.

> **Принцип «числа из кода, слова из LLM» (TOP-приоритет, второе мнение Fable).**
> Все формальные вычисления — `payback_days` (п.3), `screen_pass` (п.2),
> EV taker/maker (п.4) — считает детерминированный S1-код, а не LLM. LLM только
> **ранжирует и объясняет** на готовых числах (как `reward_trend_Nd` в §1.3).
> Если модель всё же вернула расчётную цифру — S1 пересчитывает и валидирует;
> расхождение → флаг, число оператору как факт не показывается. Причина: неверная
> цифра рядом с кнопкой «Подтвердить» — кратчайший путь от галлюцинации к потере
> денег, и это единственный барьер на нём.
> **Untrusted input:** `question`/`title` рынков — user-generated текст с
> Polymarket, попадающий в промпт = вектор prompt-injection. Экранировать/маркировать
> как данные, не инструкции; никогда не давать LLM-тексту из этих полей влиять на
> `action_type`.

| # | Проверка | Что требуется от LLM (не констатация) | Данные |
|---|---|---|---|
| 1 | **Причина деградации reward/market** | Атрибутировать падение reward_usd к драйверу: рост competitor_weight / падение est_share (разбавление), падение hours_both (односторонность), срез reward_pool_daily_rate, недодеплой capital_usd. Назвать драйвер + числа | farming_state + candidates |
| 2 | **Скрининг held-позиций** | Прогнать текущие критерии (thin_book, dead_book, days_to_end, rewards_max_spread, порог доходности — **пересчитанной по §1.5**, не из колонки) против удерживаемых рынков; отметить те, что сегодня НЕ прошли бы скрин | positions ∩ candidates |
| 3 | **PnL при кандидате на ротацию** | Для рынка под ротацию: cash_pnl/realized_pnl; при минусе — payback (сколько дней reward_usd покрывают убыток); не ротировать из минуса без обоснования payback | positions + farming_state |
| 4 | **taker/taker-both vs maker/⅓-reward** | Сравнить EV: стоимость пересечения спреда (taker обе стороны ради reward) vs пассивный maker с ⅓ reward, при текущих spread/pool | candidates (spread,pool) + farming_state |
| 5 | **fee-правило H\* ≤ 5–7 дн** | Где fees_enabled и days_to_end мал — держать горизонт ≤ 5–7 дн, чтобы fees не съели reward; отметить нарушения | candidates (fees_enabled,days_to_end) + fees_usd |

### 2.1 Формат выходного отчёта (LLM → JSON)

```
{
  "summary_ru": "<3-5 строк: главный вывод дня>",
  "per_market": [ {market, verdict_ru, screen_pass: bool, flags: [...]} ],
  "recommended_actions": [
     {id: "<uuid8>", market, action_type: hold|rotate_out|rotate_in|adjust_capital|pause_review,
      rationale_ru, confidence: low|med|high, payback_days: <n|null>}
  ]
}
```
- все human-поля — на русском (как в прецеденте);
- `recommended_actions` ранжированы по важности; каждый несёт `id` — он же
  ключ идемпотентности кнопок (§4);
- S1 рендерит из JSON человекочитаемый TG-текст + кнопки под каждым action.

## 3. OpenRouter-слой

### 3.1 Прецедент (переиспользуем, не изобретаем)

`scripts/run_weekly_whale_analysis.py` уже: грузит `.env`, читает
`OPENROUTER_API_KEY`, шлёт `{system, user-JSON}` через urllib на provider_url,
берёт `model`/`ai_provider_url` из таблицы `strategy_config` (value_text),
парсит ответ со срезанием markdown-обёртки. **Аудит наследует этот слой.**

### 3.2 Гэпы прецедента, которые спека закрывает

| Гэп в прецеденте | Решение для аудита |
|---|---|
| Нет фолбэка (одиночный вызов, raise) | retry×1 на том же model → фолбэк на вторичный model_id → при полном отказе TG-нотис «аудит недоступен» (никогда не молча) |
| Нет JSON-mode | предпочесть модель/провайдер с `response_format: json_schema`; markdown-strip-парсер оставить как fallback-парсинг |
| Нет бюджет-капа | per-run token-ceiling + месячный счётчик в таблице; при превышении — abort + нотис |

### 3.3 Выбор модели

Критерии: (контекст — не лимитирует, вход ~12k), (цена — 1 запуск/день × ~14k
токенов = центы; средний тир достаточен), (JSON-mode — обязателен). Конкретный
`ai_model` хранить в `strategy_config` (как прецедент) — сменa модели без
редеплоя. Первичная + вторичная (фолбэк) модель — две записи конфига.

### 3.4 Ключ и секреты

`OPENROUTER_API_KEY` уже в `.env` на S1 — **новый секрет не заводим**. Паттерн:
`load_dotenv()` → `os.environ.get`. Ключ БД по модели доступа — не критичен
([[project_db_access_model]]), OpenRouter-ключ — критичен, в вывод/логи не писать.

## 4. TG inline-кнопки

### 4.1 Предлагаемые действия (под каждым recommended_action)

`[✅ Подтвердить → TASK PACK]` · `[⏸ Отложить]` · `[🔍 Деталь]` · `[❌ Отклонить]`

- **Подтвердить** → фаза 3: генерит черновик TASK PACK (не исполняет);
- **Отложить** → помечает proposal deferred, вернётся в следующем аудите;
- **Деталь** → бот шлёт расширенное обоснование по этому market;
- **Отклонить** → закрывает proposal.

### 4.2 Архитектура callback: расширять бота, не плодить сервис

**Расширять `executor/farming_control_bot.py`, НЕ отдельный сервис.** Причина
техническая, не вкусовая: бот — единственный потребитель getUpdates на общем с
демоном токене («Only ONE process can use getUpdates at a time per token» —
шапка файла). Второй listener на том же токене крадёт апдейты. Отдельный сервис
потребовал бы отдельный bot-token = новый секрет + новый процесс на S2 — не
оправдано. `tg_request()` уже générичен (любой Bot API метод), поэтому в
самописный цикл добавляются: разбор `update["callback_query"]`,
`answerCallbackQuery`, `reply_markup.inline_keyboard`.

### 4.3 Идемпотентность (переиспуем существующий паттерн)

В боте уже есть `PendingConfirm` (60-сек TTL, threading.Lock) для /stop /start —
расширяем этот механизм:
- `callback_data = "audit:<action>:<proposal_id8>"` (укладывается в лимит 64 байта;
  8 hex = 32 бита, при ~5 proposal/день коллизии нереальны — достаточно, при INSERT
  проверять уникальность);
- **единственный эффект нажатия — атомарный условный переход**
  `UPDATE farming_audit_proposal SET status='confirmed' WHERE id=? AND status='pending'`;
  повтор — no-op по построению. Боту запрещено делать что-либо, кроме перехода
  статуса; материализация — отдельным шагом (Ф3, §7);
- in-memory множество обработанных id — **только UX** для быстрого
  `answerCallbackQuery("уже обработано")`; на нём НЕ держится корректность (при
  рестарте бота Telegram ре-доставляет неподтверждённые callback_query — replay
  гарантирован; защищает именно атомарный UPDATE, а не множество);
- **TTL proposal (`expires_at`, обяз.):** до следующего аудит-рана. Нажатие после
  истечения → «протухло, данные устарели», без эффекта. Иначе оператор нажмёт
  «Подтвердить» через сутки по устаревшему рынку.
- **Замечание к ветке B (§5.2):** там у бота нет прямого доступа к таблице —
  «источник правды в таблице» и атомарный UPDATE обеспечиваются только если
  определён транспорт S2→S1 (см. §5.2); без него идемпотентность в ветке B не
  обеспечена ничем.

### 4.4 Whitelist

Переиспуем `ALLOWED_CHAT_ID` (единственный chat_id). Для callback_query
проверять `callback_query.message.chat.id` И `callback_query.from.id` против
whitelist — нажатие из чужого чата игнорировать (как сейчас текстовые команды).

## 5. Оркестрация и размещение

### 5.1 Топология (следствие §0.1)

- **Producer (сбор + LLM + отправка) = S1.** Cron после снапшотов (12:35/13:35
  UTC), рядом с Postgres и OPENROUTER_API_KEY, повторяет паттерн прецедента.
  Отправляет в TG отчёт + кнопки через sendMessage+reply_markup (исходящий, без
  getUpdates).
- **Callback-приём (нажатия) = S2 bot.** Расширенный control-bot ловит
  callback_query из своего единственного getUpdates-потока.

### 5.2 Канал S1↔S2 — открытый вопрос, гейт фазы 2

Producer на S1 пишет предложения в таблицу `farming_audit_proposal` (Postgres
S1). Боту на S2 при нажатии нужно прочитать/обновить proposal. **Достижим ли
Postgres S1 с S2 — не подтверждено** (БД internal-only). Два пути, решаются в
фазе 2:
- **(A) S2→S1:5433 открыт** → бот читает/пишет proposal напрямую. Но это
  **отменяет действующую посылку безопасности** «утечка пароля БД не критична,
  т.к. БД internal-only» ([[project_db_access_model]]): открытие 5433 для S2
  расширяет периметр. Допустимо ТОЛЬКО при: (1) отдельной DB-роли с правами
  `SELECT, UPDATE(status)` на одну таблицу `farming_audit_proposal` (не общий
  postgres-суперюзер); (2) `pg_hba.conf` строго по IP S2; (3) зафиксированном
  пересмотре модели доступа в docs. При этих оговорках A предсказуемее B.
- **(B) прямого доступа нет** → нужен **явно определённый транспорт S2→S1**
  (обратного канала по умолчанию НЕТ: getUpdates монополизирован ботом на S2,
  SSH закрыт). Минимальный вариант — authed HTTPS-endpoint на S1, куда бот
  пушит только `{proposal_id, action}`; S1 выполняет атомарный переход и
  материализацию. **Ветку B нельзя принимать без определения транспорта** —
  иначе «статус переводится» некуда и идемпотентность не обеспечена (§4.3).

Рекомендация: в фазе 2 проверить достижимость (A) и решить A vs B **как money-adjacent
с confirmed-гейтом** (открытие порта БД наружу — операторское решение). Это первый
чек фазы 2, до любого кода бота.

### 5.3 n8n — не нужен

Пайплайн линейный, раз в день: собрать → вызвать LLM → отправить. Нет
ветвлений, многошаговых ретраев или внешних вебхуков, оправдывающих n8n. Прямой
cron-скрипт проще: прецедент `run_weekly_whale_analysis.py` — ровно такой cron
без n8n и работает. n8n добавил бы всегда-живой сервис + собственную поверхность
отказа ради линейной суточной задачи. **Вывод: прямой скрипт, cron.**

## 6. Границы автоматизации (карта уровней подтверждения)

| Уровень | Кто | Что можно | Гейт |
|---|---|---|---|
| **L0 — LLM автономно** | audit-скрипт S1 | читать данные, анализировать, писать отчёт, писать proposal (append-only в свои audit-таблицы), отправлять TG | нет |
| **L1 — оператор в один тап** | нажатие кнопки | подтвердить/отложить/отклонить proposal; «подтвердить» → сгенерить **черновик** TASK PACK | нажатие whitelisted-оператора |
| **L2 — оператор полный флоу** | существующий путь | исполнение TASK PACK (деплой S2) | confirmed-гейт, без изменений |

**LLM не делает НИКОГДА (жёстко, вне любой фазы):** размещение/отмена ордеров,
рестарт демона, DDL/запись в рабочие таблицы (кроме своих audit-таблиц),
изменение crontab, изменение секретов, деплой на S2. Всё money-adjacent —
за оператором через существующие гейты. Авто-аудит только **предлагает**;
материализация в действие требует нажатия, исполнение — отдельного confirmed-шага.

### 6.1 Границы как механизм, а не конвенция (второе мнение Fable)

Разделение L0 не должно держаться на дисциплине кода. Обязательно до реализации:

- **Отдельная Postgres-роль для audit-скрипта** с `GRANT` только на `farming_audit_*`
  (не общий postgres-суперюзер). L0 «append-only в свои таблицы» — техническое
  ограничение прав, а не соглашение. Иначе скрипт физически может писать в боевые
  таблицы.
- **Материализация TASK PACK (Ф3) — детерминированный шаблон из полей proposal,
  без свободного LLM-текста в исполняемой части.** Если черновик TASK PACK
  собирается вторым LLM-вызовом, галлюцинация/инъекция становится ТЗ исполнителю,
  а оператор под automation bias её проштампует. Исполняемые поля (market, side,
  size, action_type) берутся из структурированного proposal; LLM-текст —
  только в human-обосновании, не в параметрах действия.
- **Kill switch:** флаг в `strategy_config` (напр. `audit_enabled=false`),
  который глушит авто-аудит без правки crontab (спам, сломанная модель,
  компрометация ключа). Скрипт проверяет флаг первым делом и тихо выходит.
  Crontab под confirmed-гейтом — не годится как аварийный выключатель.

### 6.2 Осознанный системный риск: automation bias

Система по построению снижает бдительность оператора (в этом её цель) — поэтому
барьеры §2 («числа из кода»), §4.3 (TTL, атомарность), §6.1 (механизм, не
конвенция) не опциональны: при нулевой защите контента LLM неверная рекомендация
рядом с кнопкой «Подтвердить» доходит до денег за одно доверчивое нажатие.

## 7. Этапность внедрения

| Фаза | Содержание | Критерий перехода |
|---|---|---|
| **Ф1 — read-only отчёт** | S1: дайджест-билдер + prune, OpenRouter-вызов (фолбэк, бюджет-кап), рендер, sendMessage. **Ноль изменений S2, ноль cross-host state.** Даёт ~80% ценности (анализ) | отчёт качественно принят оператором на протяжении N дней: вывод LLM совпадает с ручным вердиктом методологии 15.07 |
| **Ф2 — кнопки-заглушки** | Кнопки под action'ами; S2-бот ловит callback, answerCallbackQuery, **пишет решение** (пока без TASK PACK). Цель фазы — валидировать callback-плюмбинг + идемпотентность + **канал S1↔S2 (§5.2)** | нажатия надёжны, идемпотентны, канал (A|B) подтверждён; whitelist на callback работает |
| **Ф3 — TASK PACK по кнопке** | «Подтвердить» → генерит черновик TASK PACK на S1 (где данные+шаблоны); доставка в TG. Исполнение — существующий L2, LLM не исполняет | черновики TASK PACK корректны на M примерах; операторская приёмка |

Опционально после Ф3: **FARM-047** — episode-обогащение (демон S2 персистит
halt/pause/reconcile в Postgres). Вне сессии (изменение демона на S2).

## 8. Оценка и тикеты

### 8.1 Трудоёмкость (порядок)

| Тикет | Фаза | Суть | Оценка | Гейты |
|---|---|---|---|---|
| **FARM-043** | Ф1 | S1 audit-assembler + OpenRouter (фолбэк/бюджет) + read-only TG-отчёт | ~1–1.5 дня | — |
| **FARM-044** | Ф2 | inline-кнопки + S2-бот callback + идемпотентность + проверка канала S1↔S2 | ~1 день + **деплой S2 (вне сессии)** |
| **FARM-045** | Ф3 | генерация черновика TASK PACK по «подтвердить» | ~1 день | операторская приёмка |
| **FARM-046** | Ф1/2 | audit-таблицы (`farming_audit_run`, `farming_audit_proposal` c `expires_at`) + бюджет-счётчик + **отдельная DB-роль** на `farming_audit_*` + kill-switch в `strategy_config` | ~0.5 дня | **DDL/роль = confirmed-гейт** |
| **FARM-047** | опц. | episode→Postgres обогащение (демон S2) | отложено | вне сессии |

### 8.2 Порядок запуска

FARM-046 (таблицы) → FARM-043 (Ф1) → FARM-044 (Ф2) → FARM-045 (Ф3). FARM-043
можно начать без FARM-046, если Ф1 шлёт отчёт без персиста proposal (proposal
нужен с Ф2). Рекомендация: FARM-046 + FARM-043 в одном заходе (Ф1 сразу пишет
run-историю).
