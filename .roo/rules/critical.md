 # 🚨 КРИТИЧЕСКИЕ ПРАВИЛА — ОБЯЗАТЕЛЬНЫ К ИСПОЛНЕНИЮ

Нарушение любого пункта считается серьёзной ошибкой.
Эти правила имеют приоритет выше всех остальных инструкций.

---

# ❌ ЗАПРЕЩЕНО

## 1. НЕ перезаписывать рабочие конфиги напрямую

❌ Нельзя:
echo "new config" > requirements.txt

✅ Правильно:
1. Создать backup
2. Создать новый файл
3. Протестировать
4. Только потом заменить

Рабочая система должна оставаться рабочей.

---

## 2. НЕ делать 3+ неудачных сборки подряд

Если сборка docker провалилась:
1. ОСТАНОВИТЬСЯ
2. Проанализировать причину
3. Исправить root cause
4. Только потом повторить

Повторять без анализа запрещено.

---

## 3. НЕ трогать /var/lib/docker вручную

Запрещено:
- rm -rf /var/lib/docker/*
- chmod 777 /var/lib/docker
- удалять overlay2 напрямую

Использовать только:
- docker system prune
- docker volume rm
- docker rmi

Ручные изменения ломают внутреннее состояние Docker.

---

## 4.  НЕ удалять контейнеры без проверки принадлежности к polymarket-bot

Перед удалением любого контейнера или volume:

1. Проверить labels:
docker inspect <container> --format '{{.Config.Labels}}'

2. Убедиться, что контейнер относится к docker-compose проекту polymarket-bot:
   label com.docker.compose.project=polymarket-bot

3. Если label отсутствует — остановиться и уточнить у пользователя.

Удалять контейнеры, не относящиеся к polymarket-bot, запрещено.
Удаление “всего подряд” запрещено.
---

## 5. НЕ работать над чужими проектами

Если на сервере есть другие сервисы (например n8n):
- Не вмешиваться
- Сообщить пользователю
- Работать только в границах polymarket-bot

---

## 6. НЕ начинать работу без чтения контекста

Перед любой нетривиальной задачей:
1. Прочитать memory-bank/projectBrief.md
2. Прочитать memory-bank/errors-log.md
3. Прочитать затрагиваемые файлы полностью

Работа без контекста запрещена.

---

## 7. НЕ удалять volume БД без backup

Перед любым удалением volume:

1. Проверить наличие данных
2. Сделать pg_dump
3. Убедиться в сохранности бэкапа

Удаление volume без резервной копии запрещено.

---

## 8. НЕ игнорировать результат команд

После каждой команды:
- Проверить код возврата
- Проверить фактический результат
- Убедиться, что состояние соответствует ожиданию

Предположения запрещены.

## 9. 🔐 СЕКРЕТЫ (ОБЯЗАТЕЛЬНО)

### Запрещено показывать или выводить:
- содержимое `.env` и любых файлов с секретами
- приватные ключи (private key / seed / mnemonic)
- API keys, токены, пароли, ключи доступа
- любые значения из переменных окружения, содержащие `KEY`, `SECRET`, `TOKEN`, `PASSWORD`

### Разрешено только в замаскированном виде:
- показывать факт наличия переменной (set / not set)
- показывать последние 4 символа (пример: `****a1b2`)
- показывать длину значения (пример: `len=64`)

### Безопасные проверки вместо вывода:
- `printenv | grep -E "KEY|SECRET|TOKEN|PASSWORD"` — только имена переменных (без значений)
- `test -n "$VAR" && echo "VAR: set" || echo "VAR: not set"`
---

## 10. 🚫 НЕ выполнять git commit / git push без явного подтверждения STRATEGY

git commit и git push — **запрещены** без явной фразы подтверждения от пользователя.

❌ Нельзя:
- коммитить после завершения задачи "по умолчанию"
- считать успешный результат достаточным основанием для коммита
- коммитить "для сохранения прогресса"

✅ Обязательно:
- дождаться явного "confirmed: commit" или "коммить" от STRATEGY
- показать предлагаемый commit message ДО выполнения
- только после одобрения — выполнить git commit

Нарушение зафиксировано дважды: SYS-325, SYS-333.

# 🛡 ПРОТОКОЛЫ БЕЗОПАСНОСТИ

## Перед удалением ресурса

1. Проверить ownership (labels)
2. Проверить зависимости
3. Проверить размер/важность
4. Если есть сомнения — спросить пользователя

---

## Перед сборкой Docker

1. Проверить свободное место (df -h)
2. Убедиться, что достаточно минимум 2–3GB
3. Проверить существующие образы
4. Спланировать очистку cache после сборки

---

## Перед git push

1. Проверить изменения (git status, git diff)
2. Проверить отсутствие секретов
3. Протестировать локально
4. Сделать понятный commit message

---

# 🔍 ОБЯЗАТЕЛЬНЫЕ ПРОВЕРКИ КАЖДОЙ СЕССИИ

1. Проверить свободное место (df -h)
2. Проверить контейнеры проекта
3. Ознакомиться с projectBrief
4. Ознакомиться с errors-log

Если любой пункт пропущен — работа не начинается.
