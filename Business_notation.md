ШАГ 1. ПОДКЛЮЧЕНИЕ К POLYMARKET DATA API
1. Назначение шага
Шаг обеспечивает получение рыночных данных Polymarket по адресам отслеживаемых трейдеров для последующего запуска pipeline копирования сделок.
Цель шага:
установить HTTP-подключение к Polymarket Data API
запросить сделки и, при необходимости, позиции по адресу кита
преобразовать ответ API во внутренний формат системы
передать данные дальше в execution pipeline
Базовый URL API зафиксирован в WhaleTracker.DATA_API_URL как https://data-api.polymarket.com . В состоянии проекта также отдельно зафиксировано, что источником данных является именно Polymarket Data API, API key не требуется, источник считается валидным и рабочим .

2. Исходный файл
Основной файл шага:
whale_tracker.py
Отвечает за создание HTTP-сессии, вызовы /trades и /positions, обработку ответа API и приведение данных к внутренним dataclass-структурам WhaleTrade и WhalePosition .
Файл-инициатор в текущем paper execution path:
main.py
Создаёт экземпляр WhaleTracker, подключает БД, загружает список китов и периодически вызывает fetch_whale_trades() для выбранных адресов .

3. Контейнер
Для текущего execution path этот шаг выполняется в контейнере:
polymarket_bot
Обоснование:
main.py является точкой входа paper execution loop
именно он инициализирует WhaleTracker и запускает регулярные API-вызовы .

Дополнительно:

в проекте также существует отдельный сервис whale_detector, и в PROJECT_STATE.md зафиксировано, что Polymarket Data API используется и в whale discovery pipeline, который активен и работает .
То есть архитектурно Data API используется не только в polymarket_bot, но для данного шага в контексте paper copy execution основной зафиксированный runtime-путь идёт через polymarket_bot.

4. Интервал выполнения
В текущей реализации цикл работает постоянно:
основной цикл while True
итерация каждую 1 секунду
реальный запрос к сделкам китов выполняется каждые 5 итераций, то есть примерно каждые 5 секунд (check_interval = 5) .
На каждом цикле проверки:
выбираются первые 5 китов из загруженного списка
по каждому киту запрашиваются последние 20 сделок
из них в обработку идут первые 3 сделки .

5. Алгоритм шага
5.1. Инициализация клиента
main.py создаёт объект WhaleTracker(database_url=database_url)
вызывается whale_tracker.set_database(database_url)
при первом HTTP-вызове внутри WhaleTracker создаётся aiohttp.ClientSession() через _ensure_http_session() .

5.2. Формирование запроса
Для получения сделок вызывается:
GET /trades?user=0xADDRESS&limit=N
Для получения позиций предусмотрен отдельный вызов:
GET /positions?user=0xADDRESS .
Параметры:
user = адрес кита в lower-case
limit = число возвращаемых сделок, по умолчанию 100, в main.py используется 20 .

5.3. Таймаут и транспортный режим
Запросы выполняются через aiohttp, с таймаутом:
aiohttp.ClientTimeout(total=30) .

5.4. Обработка ответа
Если HTTP-статус не 200:
пишется warning в лог
возвращается пустой список
выполнение pipeline для данного адреса фактически пропускается на текущем цикле .
Если статус 200:

JSON-ответ читается через resp.json()

каждая запись приводится к внутреннему объекту WhaleTrade или WhalePosition

timestamp нормализуется:

None → datetime.now()

int/float → datetime.fromtimestamp(...)

ISO string → datetime.fromisoformat(...)

ошибочный формат → fallback на datetime.now() .

Шаг считается успешным, если:

JSON успешно распарсен

записи успешно преобразованы во внутренние объекты

список сделок/позиций передан дальше в pipeline .

Частично успешное выполнение

Шаг считается частично успешным, если:

подключение есть

ответ получен

часть записей содержит проблемный timestamp, но данные нормализованы через fallback на datetime.now() .

Неуспешное выполнение

Шаг считается неуспешным для конкретного адреса, если:

HTTP status ≠ 200

произошёл timeout

возникло исключение при запросе или парсинге
В этих случаях возвращается пустой список, а цикл продолжает работу по другим адресам .

10. Особые случаи
10.1. API не даёт settlement/outcome-метрики

В коде прямо зафиксировано, что Polymarket Data API не предоставляет надёжные данные для win_rate и realized PnL трейдера. Поэтому шаг подключения к API не может сам по себе дать:

winrate кита

подтверждённый profit китa

settlement status по адресной истории трейдера .

Следствие:

на этом шаге можно получить только activity data и snapshot data

оценка edge строится дальше на paper trading результатах, а не на “магическом знании” API о доходности кита. Мир, к счастью, не настолько щедр.

10.2. Нормализация времени

API может вернуть timestamp в разных форматах:

Unix

ISO string

пустое значение
Код обрабатывает все варианты и не ломает pipeline из-за одного кривого поля времени .

10.3. Нестабильный или неполный ответ API

При любой ошибке по одному киту:

весь бот не падает

ошибка логируется

цикл продолжается по следующим адресам и на следующей итерации повторит попытку .

10.4. Текущая реализация поля размера сделки

В загруженной версии whale_tracker.py в fetch_whale_trades() размер берётся из item.get("amount", 0) .
При этом ранее в проекте уже была зафиксирована проблема, что API возвращает size, а не amount, из-за чего возникали zero-size trades. Это важно для верификации шага, потому что технически подключение к API работает, но корректность маппинга поля размера относится к критическим условиям валидности данных в следующем шаге pipeline. Это отражено в проектной истории и аудите таблицы trades .

11. Результат шага

Результатом шага является:

активное подключение к Polymarket Data API

получение актуальных сделок/позиций по адресам китов

преобразование ответа в унифицированный внутренний формат

передача данных в следующий слой pipeline для дедупликации и paper-исполнения .

12. Краткая бизнес-формула шага

polymarket_bot
→ инициализирует WhaleTracker
→ создаёт HTTP session
→ обращается к https://data-api.polymarket.com/trades и .../positions
→ получает JSON по адресу кита
→ нормализует поля и timestamp
→ формирует WhaleTrade / WhalePosition
→ передаёт данные в следующий шаг pipeline .

# ШАГ 2 — ФОРМИРОВАНИЕ И ОБНОВЛЕНИЕ ТАБЛИЦЫ `whales`

## 1. Назначение шага

Шаг отвечает за формирование рабочего списка китов, которые допускаются
в систему copy trading.

На этом шаге система не просто сохраняет любой замеченный адрес, а
выполняет **qualification** на основе уже собранных сделок в таблице
`whale_trades`.

Цель шага:

1. проанализировать накопленные сделки из `whale_trades`
2. агрегировать торговую активность по адресам трейдеров
3. определить, какие адреса соответствуют критериям "кита"
4. записать новых квалифицированных трейдеров в таблицу `whales`
5. обновить статистику и статус уже существующих китов
6. поддерживать актуальный список адресов для downstream pipeline

Результатом шага является таблица `whales`, содержащая
**квалифицированных трейдеров**, пригодных для дальнейшего copy-trading pipeline.

---

## 2. Место шага в общем pipeline

Шаг находится **после записи сделок в `whale_trades`** и **до использования
адресов системой копирования**.

Бизнес-цепочка:

`WebSocket / ingestion`
→ `whale_trades`
→ `qualification`
→ `whales`
→ `copy pipeline`
→ `paper_trades`
→ `trades`

Это означает, что таблица `whales` является не сырым реестром адресов,
а **результатом аналитического отбора** по данным из `whale_trades`.

---

## 3. Исходный файл

Основной файл шага:

- `whale_tracker.py`

В рамках данного шага файл отвечает за логику:

- чтения исторических или накопленных whale trades
- агрегации статистики по адресам
- qualification трейдеров
- записи результата в таблицу `whales`

Если в коде qualification размазан по нескольким функциям, то бизнесово
это всё равно считается **единым алгоритмом отбора китов**.

---

## 4. Контейнер

Контейнер выполнения:

- `whale_detector`

Назначение контейнера:

- сбор и обработка потока данных по сделкам китов
- наполнение `whale_trades`
- вычисление qualification
- формирование актуального списка китов в `whales`

Контейнер `whale_detector` является владельцем этого шага.

---

## 5. Интервал выполнения

Шаг выполняется периодически внутри сервиса `whale_detector`.

Модель выполнения:

1. сервис получает новые сделки
2. сделки сохраняются в `whale_trades`
3. затем выполняется qualification по накопленным данным
4. таблица `whales` обновляется

Бизнесово интервал шага зависит от цикла работы `whale_detector`.

Если описывать без привязки к конкретной секунде:
- шаг выполняется **регулярно**
- шаг может запускаться **после каждой порции новых сделок**
- шаг может работать как **batch refresh** по расписанию

Для технической документации важно фиксировать, что qualification
не является ручным действием и не выполняется ad hoc, а входит
в регулярный автоматический pipeline контейнера `whale_detector`.

---

## 6. Входные данные шага

Источник входных данных:

- таблица `whale_trades`

Шаг использует уже записанные в БД сделки китов.

Минимально значимые входные атрибуты:

- адрес трейдера
- идентификатор рынка
- направление сделки
- размер сделки
- цена
- timestamp сделки

Именно эти данные позволяют агрегировать активность трейдера
и оценивать его пригодность для попадания в список китов.

---

## 7. База данных, используемая шагом

### Таблица-источник

- `whale_trades`

Назначение:
хранение обнаруженных сделок трейдеров, поступивших из Polymarket.

### Целевая таблица

- `whales`

Назначение:
хранение квалифицированных адресов трейдеров, допущенных
в систему дальнейшего copy-trading анализа и исполнения.

---

## 8. Бизнес-логика шага

### 8.1. Чтение накопленных сделок

Система извлекает из `whale_trades` набор сделок для анализа.

Задача этапа:
- определить, какие адреса проявляют достаточно крупную и/или устойчивую активность
- получить базу для агрегирования статистики по каждому трейдеру

На этом этапе данные ещё не являются списком китов.
Это только сырой торговый поток, накопленный в `whale_trades`.

---

### 8.2. Группировка по адресу трейдера

Сделки группируются по wallet address.

Для каждого адреса рассчитываются агрегаты, например:

- количество сделок
- суммарный объём
- средний размер сделки
- частота активности
- число уникальных рынков
- время последней активности

Бизнесовый смысл этапа:
система должна оценивать не одну отдельную сделку, а
**поведение адреса на накопленном интервале наблюдения**.

---

### 8.3. Qualification трейдера

После агрегации система применяет правила qualification.

Из подтверждённого проектного состояния следует, что qualification
выполняется на основе `whale_trades`, а результатом является
**top 50 whales for copying**.

Следовательно, на этом шаге используются критерии отбора,
основанные как минимум на накопленной торговой активности.

Бизнесово qualification включает:

1. исключение адресов с недостаточной активностью
2. ранжирование адресов по значимому показателю
3. отбор ограниченного числа лучших адресов
4. присвоение адресам статуса пригодных для копирования

Согласно проектному состоянию системы, итоговый рабочий список:
- **Top 50 whales**
- критерий отбора: **qualified / auto_detected**
- ranking: **by volume**

Это означает, что ключевой квалификационный признак шага:
**адрес должен входить в top 50 по объёму на основе данных `whale_trades`**

---

### 8.4. Проверка существования записи в `whales`

Для каждого адреса, прошедшего qualification, система проверяет,
существует ли он уже в таблице `whales`.

Возможны два сценария:

1. адреса в таблице ещё нет
2. адрес уже существует и требует обновления

---

### 8.5. Создание новой записи в `whales`

Если квалифицированный адрес отсутствует в таблице `whales`,
создаётся новая запись.

Бизнесовый смысл:
адрес впервые признан системой достаточно значимым,
чтобы участвовать в дальнейшем copy-trading pipeline.

При создании записи обычно фиксируются:

- wallet address
- время первого включения в список китов
- время последней активности
- агрегированная статистика на момент qualification
- статус активности
- признаки qualification / ranking

Даже если конкретные имена полей отличаются,
бизнесово запись означает:

**"данный адрес признан системой китом и допущен к дальнейшему отслеживанию"**

---

### 8.6. Обновление существующей записи в `whales`

Если адрес уже присутствует в таблице `whales`,
система обновляет его статистику и актуальное состояние.

Обновляться могут:

- время последней активности
- накопленный объём
- число сделок
- ranking position
- qualification status
- внутренние служебные поля для отбора

Бизнесовый смысл:
таблица `whales` должна отражать **актуальное состояние кита**,
а не только факт его первичного появления в системе.

---

### 8.7. Исключение адресов, не прошедших qualification

Если адрес присутствует в `whale_trades`, но не проходит qualification,
возможны два бизнес-сценария:

1. адрес не добавляется в `whales`
2. адрес, ранее бывший китом, помечается как неактуальный /
   теряет активный статус / выбывает из top-50

Как именно это реализовано физически, зависит от схемы таблицы,
но бизнесово важно зафиксировать:

таблица `whales` не обязана содержать всех когда-либо замеченных трейдеров;
она должна содержать **актуальный квалифицированный набор**.

---

## 9. Формат записи в таблицу `whales`

Таблица `whales` хранит не сырые сделки, а **сводную карточку трейдера**.

Типы данных, которые бизнесово должны присутствовать в записи:

- адрес трейдера
- признак, что адрес квалифицирован как кит
- агрегированная статистика торговли
- время первой фиксации
- время последней активности
- текущий статус пригодности для copy trading

Таким образом, одна запись в `whales` соответствует
**одному трейдеру**, а не одной сделке.

---

## 10. Записи в БД на этом шаге

На шаге выполняются операции двух типов:

### 10.1. INSERT

Используется, если:
- адрес прошёл qualification
- в таблице `whales` его ещё нет

Результат:
- создаётся новая запись о ките

### 10.2. UPDATE

Используется, если:
- адрес уже есть в `whales`
- появились новые данные из `whale_trades`
- изменился рейтинг, объём, активность или qualification state

Результат:
- существующая запись приводится к актуальному состоянию

---

## 11. Статусы и условия

Шаг должен учитывать как минимум следующие бизнес-состояния адреса:

### 11.1. Новый квалифицированный кит
Адрес ранее не был в `whales`, но теперь прошёл qualification.

Действие:
- INSERT в `whales`

### 11.2. Действующий кит
Адрес уже присутствует в `whales` и сохраняет qualification.

Действие:
- UPDATE статистики и служебных атрибутов

### 11.3. Выбывший кит
Адрес ранее был в рабочем пуле, но перестал входить в актуальный
qualified set / top-50.

Действие:
- либо снимается active status
- либо запись остаётся исторически, но перестаёт использоваться downstream

### 11.4. Неквалифицированный трейдер
Адрес встречается в `whale_trades`, но не соответствует правилам отбора.

Действие:
- не попадает в рабочий список `whales`

---

## 12. Особые случаи

### 12.1. Адрес есть в `whale_trades`, но не должен копироваться
Это нормальная ситуация.

Причина:
`whale_trades` содержит более широкий поток наблюдений,
а `whales` содержит только квалифицированную подвыборку.

---

### 12.2. Адрес временно выпадает из top-50
Если qualification строится на ranking by volume,
адрес может выйти из рабочего списка без каких-либо ошибок в системе.

Это не сбой, а нормальное поведение qualification layer.

---

### 12.3. Дубликаты адресов
Таблица `whales` должна допускать только одну актуальную запись
на один wallet address.

Бизнесовое правило:
один трейдер = одна master-запись в `whales`.

---

### 12.4. Неполные данные в `whale_trades`
Если часть сделок содержит неполные или проблемные значения,
это может влиять на ranking и qualification.

Следствие:
качество шага 2 напрямую зависит от корректности ingestion в `whale_trades`.

Именно поэтому этот шаг нельзя описывать в отрыве от шага записи
в `whale_trades`. Чудес не бывает, к сожалению.

---

### 12.5. Исторический след
Даже если адрес перестал быть активным китом,
система может сохранять его запись для истории, аудита и анализа.

Это особенно важно для:
- объяснимости qualification
- повторной оценки адресов
- исследования качества отбора

---

## 13. Результат шага

Результатом шага является:

1. актуализированная таблица `whales`
2. сформированный qualified set адресов
3. рабочий список top-50 китов для последующего copy trading pipeline

Итог бизнесово означает:

система не копирует сделки любого встреченного адреса,
а работает только с адресами, которые уже прошли qualification
по данным из `whale_trades`.

---

## 14. Использование результата в следующем шаге

Результат шага используется downstream-процессами как источник
разрешённых адресов для копирования.

Следующий бизнес-шаг:

- отбор сделок только по qualified whales
- формирование `paper_trades`
- дальнейшее paper execution

Иными словами:

`whales` — это фильтр допуска в copy-trading pipeline.

---

## 15. Краткая бизнес-формула шага

`whale_detector`
→ читает накопленные данные из `whale_trades`
→ агрегирует статистику по wallet address
→ выполняет qualification
→ отбирает top-50 whales by volume
→ создаёт или обновляет записи в `whales`
→ передаёт qualified set в downstream copy pipeline


# ШАГ 3 — ЗАПИСЬ СДЕЛОК В ТАБЛИЦУ whale_trades

# ШАГ 4 — QUALIFICATION КИТОВ И ФОРМИРОВАНИЕ ТАБЛИЦЫ whales

## 1. Назначение шага

Шаг отвечает за qualification трейдеров и формирование рабочего списка китов в таблице whales.

На этом шаге система:

1. читает накопленные сделки из whale_trades
2. рассчитывает агрегированные метрики по каждому адресу
3. применяет qualification criteria
4. определяет qualification path
5. обновляет таблицу whales
6. формирует набор адресов, доступных downstream для copy trading

Результатом шага является таблица whales, содержащая квалифицированных и ранжированных трейдеров, пригодных для дальнейшего использования в системе.

---

## 2. Место шага в pipeline

Общая бизнес-цепочка:

Polymarket  
→ ingestion  
→ whale_trades  
→ qualification  
→ whales  
→ paper_trades  
→ trades  
→ settlement  
→ bankroll

Этот шаг преобразует сырые сделки из whale_trades в структурированный список трейдеров whales.

---

## 3. Контейнер выполнения

Контейнер:

whale-detector

Entrypoint:

python src/run_whale_detection.py

Именно этот сервис выполняет:

- qualification по новым сделкам
- периодическое обновление qualification
- обновление activity counters
- обновление записей в whales

---

## 4. Исходный файл

Основной файл:

src/research/whale_detector.py

Ключевые функции:

- _evaluate_quality()
- _calculate_qualification_path()
- refresh_qualification()
- update_whale_activity_counters()
- get_top_whales()
- _save_whale_to_db()

Downstream использование результата:

src/research/whale_tracker.py  
функция load_quality_whales()

---

## 5. Интервал выполнения

Qualification выполняется в нескольких режимах.

### 5.1 Inline qualification

Метод _evaluate_quality() вызывается при обработке каждой новой сделки.

Это означает, что первичная qualification может происходить сразу на входящем trade event.

### 5.2 Periodic refresh

Методы refresh_qualification(), update_whale_activity_counters() и get_top_whales() выполняются периодически.

Интервал:

примерно 1 раз в час

### 5.3 Startup load

Результаты qualification затем загружаются при старте downstream-компонентов через load_quality_whales().

---

## 6. Источник данных

Основной источник данных:

таблица whale_trades

Для qualification используются данные по адресам трейдеров, сохранённые в whale_trades.

Ключевые поля, используемые в расчётах:

- wallet_address
- size_usd
- traded_at
- market_id

Дополнительно qualification использует уже существующие записи whales для periodic refresh и повторного расчёта активности.

---

## 7. Алгоритм шага

## 7.1 Чтение сделок из whale_trades

Система агрегирует сделки по wallet_address.

Используются отдельные запросы для расчёта активности на разных окнах:

- trades_last_3_days
- trades_last_7_days
- days_active

Смысл этапа:

получить статистический профиль каждого трейдера на основе его накопленных сделок.

---

## 7.2 Расчёт activity counters

Система вычисляет следующие показатели.

### trades_last_3_days

Количество сделок адреса за последние 3 дня.

### trades_last_7_days

Количество сделок адреса за последние 7 дней.

### days_active

Количество уникальных дней активности адреса.

Важно:

days_active считается по всей доступной истории, а не по ограниченному временному окну.

---

## 7.3 Использование базовых метрик трейдера

Для каждого трейдера система использует или обновляет следующие агрегаты:

- total_trades
- total_volume_usd
- avg_trade_size_usd
- win_rate
- total_profit_usd
- risk_score
- trades_last_3_days
- trades_last_7_days
- days_active

Часть этих значений рассчитывается из whale_trades, часть хранится и обновляется в таблице whales.

---

## 7.4 Binary qualification gate

Система применяет обязательные qualification criteria.

Трейдер считается qualified только если одновременно выполняются все условия:

- total_trades >= 10
- trades_last_3_days >= 3
- total_volume_usd >= 500
- days_active >= 1

Это жёсткий бинарный фильтр.

Если хотя бы одно условие не выполнено, трейдер не проходит qualification.

---

## 7.5 Определение qualification path

После прохождения базового qualification система определяет qualification path.

Используются два пути.

### ACTIVE path

Трейдер относится к ACTIVE path, если выполняются условия:

- total_trades >= 10
- total_volume_usd >= 500
- trades_last_7_days >= 3
- risk_score <= 6

Этот путь соответствует активному и регулярно торгующему адресу.

### CONVICTION path

Трейдер относится к CONVICTION path, если выполняются условия:

- total_volume_usd >= 10000
- avg_trade_size_usd >= 2000
- risk_score <= 6

Этот путь соответствует адресу с крупными и редкими, но значимыми сделками.

Таким образом система допускает два типа китов:

- активные систематические трейдеры
- крупные conviction-трейдеры

---

## 7.6 Обновление qualification status

После qualification система присваивает или обновляет атрибуты записи в whales.

Используются как минимум следующие поля:

- status
- qualification_path
- last_qualified_at
- last_ranked_at
- updated_at
- is_active

Поле status имеет значения:

- discovered
- qualified
- ranked

Бизнесовый смысл:

discovered — адрес обнаружен системой  
qualified — адрес прошёл qualification  
ranked — адрес вошёл в рабочий ranking

---

## 7.7 Ranking qualified whales

После qualification система ранжирует qualified-трейдеров по score formula.

Используемые компоненты score:

- risk_component
- activity_component
- volume_component

Логика ranking:

risk_component  
чем ниже risk_score, тем выше рейтинг

activity_component  
чем больше trades_last_3_days, тем выше рейтинг

volume_component  
чем больше total_volume, тем выше рейтинг

Итоговый rank_score используется для сортировки qualified whales по убыванию.

---

## 7.8 Ограничение по top N

После ranking система возвращает ограниченный набор лучших трейдеров.

По умолчанию:

- get_top_whales() использует limit = 10
- downstream copy trading использует LIMIT 50

Следовательно, в системе одновременно существуют как минимум два уровня ограничения выборки:

- внутренний ranking layer
- downstream рабочий пул для copy trading

---

## 7.9 Запись результата в whales

После qualification и ranking выполняется UPSERT в таблицу whales.

Логика записи:

- если wallet_address ещё отсутствует, создаётся новая запись
- если wallet_address уже существует, запись обновляется

Обновляются следующие ключевые атрибуты:

- total_trades
- total_volume_usd
- avg_trade_size_usd
- risk_score
- status
- trades_last_3_days
- trades_last_7_days
- days_active
- qualification_path
- updated_at

Таким образом whales является не историческим логом сделок, а master-таблицей состояния трейдера.

---

## 8. Структура таблицы whales

Таблица whales содержит одну запись на одного трейдера.

Основные поля:

id — SERIAL — первичный ключ

wallet_address — VARCHAR(66) — уникальный адрес трейдера

first_seen_at — TIMESTAMP — момент первого появления

total_trades — INTEGER — общее число сделок

win_rate — DECIMAL — метрика качества трейдера

total_profit_usd — DECIMAL — агрегированный profit

total_volume_usd — DECIMAL — общий объём торговли

avg_trade_size_usd — DECIMAL — средний размер сделки

last_active_at — TIMESTAMP — последняя активность

is_active — BOOLEAN — флаг активности

risk_score — INTEGER — риск-профиль трейдера

status — VARCHAR(20) — discovered / qualified / ranked

trades_last_3_days — INTEGER — сделки за 3 дня

days_active — INTEGER — число уникальных активных дней

last_qualified_at — TIMESTAMP — время последней qualification

updated_at — TIMESTAMP — время последнего обновления

Ключевое ограничение:

wallet_address UNIQUE

Это обеспечивает одну master-запись на одного трейдера.

---

## 9. Использование результата downstream

Результат qualification используется в downstream-компонентах через выборку из whales.

Downstream фильтры при загрузке quality whales:

- is_active = TRUE
- win_rate >= min_win_rate
- total_trades >= min_trades
- risk_score <= max_risk_score

Сортировка downstream:

- сначала по win_rate DESC
- затем по total_trades DESC

Ограничение downstream:

LIMIT 50

Именно этот пул используется системой copy trading.

## 10. Особые случаи

## 10.1 NULL wallet_address

Это критический риск.

Если запись в whale_trades не содержит wallet_address, qualification не может корректно связать сделку с трейдером.

Следствие:

часть сделок может выпадать из qualification pipeline.

---

## 10.2 Дубликаты whale_trades

Если в whale_trades присутствуют дубликаты, это искажает:

- total_trades
- total_volume_usd
- activity counters
- ranking

Следовательно качество qualification напрямую зависит от качества ingestion.

---

## 10.3 Разные окна данных

Система использует смешанную модель временных окон:

- 3 дня для trades_last_3_days
- 7 дней для trades_last_7_days
- вся история для days_active

Это означает, что qualification сочетает краткосрочную активность и долгосрочную историю.

---

## 10.4 Отсутствие time decay

В текущем алгоритме нет полноценного механизма устаревания трейдера.

Это значит, что адрес может сохранять сильный исторический профиль, даже если его актуальная ценность уже снизилась.

---

## 10.5 Жёсткие лимиты

В системе есть hardcoded limits:

- ranking limit = 10
- downstream copy trading limit = 50

Это создаёт риск несогласованности между внутренним ranking и фактическим рабочим пулом для копирования.

Результатом шага является актуализированная таблица whales.

Таблица содержит:

# ШАГ 5 — COPY TRIGGER И ФОРМИРОВАНИЕ ТАБЛИЦЫ paper_trades

## 1. Назначение шага

Шаг отвечает за принятие решения о копировании сделки кита и создание записи в таблице paper_trades.

На этом шаге система:

1. реагирует на новую запись в whale_trades
2. проверяет, соответствует ли трейдер условиям copy trading
3. применяет фильтры допуска
4. рассчитывает размер paper-сделки
5. выполняет дедупликацию
6. создаёт запись в paper_trades

Таблица paper_trades является staging-слоем между источником сигнала и фактическим исполнением virtual trade.

---

## 2. Место шага в pipeline

Общая бизнес-цепочка:

Polymarket  
→ ingestion  
→ whale_trades  
→ qualification  
→ whales  
→ copy trigger  
→ paper_trades  
→ execution  
→ trades  
→ settlement  
→ bankroll

Этот шаг соединяет qualification слой с execution слоем.

Именно здесь система отвечает на вопрос:

копировать ли конкретную сделку кита в paper-режиме.

---

## 3. Контейнер и место выполнения

Физически логика copy trigger выполняется внутри PostgreSQL.

То есть решение о создании paper_trade принимается:

- не в приложении Python
- не в main.py
- не в отдельном сервисе

а внутри trigger-функции базы данных.

Точка реализации:

scripts/create_copy_trigger.sql

Trigger-функция:

copy_whale_trade_to_paper()

Trigger:

trigger_copy_whale_trade

Тип trigger:

AFTER INSERT ON whale_trades

Это означает, что каждый новый INSERT в whale_trades автоматически запускает проверку на создание paper_trade.

---

## 4. Исходные файлы и компоненты

Основные компоненты шага:

scripts/create_copy_trigger.sql  
реализует copy logic, filters, sizing, deduplication и INSERT в paper_trades

src/main.py  
читает paper_trades как pending staging-слой и передаёт их в execution

Связанные ingestion-компоненты, которые порождают события для этого шага:

src/research/whale_detector.py  
src/research/whale_tracker.py  
src/research/real_time_whale_monitor.py

Они записывают сделки в whale_trades, после чего trigger в PostgreSQL запускает copy pipeline.

---

## 5. Интервал выполнения

Шаг выполняется событийно.

Модель выполнения:

1. в whale_trades вставляется новая запись
2. immediately срабатывает PostgreSQL trigger
3. trigger-функция проверяет copy conditions
4. при успешном прохождении условий создаётся запись в paper_trades

Таким образом у шага нет batch-расписания.

Он работает в режиме:

on insert into whale_trades

---

## 6. Источник данных

Непосредственный источник сигнала:

таблица whale_trades

Trigger использует новую вставленную запись NEW из whale_trades.

Дополнительно для принятия решения используются данные из:

- whales
- whale_trades
- paper_trades

Это означает, что copy trigger использует не только саму новую сделку, но и контекст трейдера и историю недавнего копирования.

---

## 7. Алгоритм шага

## 7.1 Событие запуска

Алгоритм запускается после вставки новой сделки в whale_trades.

Система получает из NEW:

- wallet / whale address
- market_id
- market_title
- side
- price
- size_usd
- traded_at

Эта запись рассматривается как кандидат на копирование.

---

## 7.2 Проверка попадания трейдера в рабочий пул

Перед созданием paper_trade система проверяет, входит ли адрес в рабочий copy universe.

Используются следующие gates.

### Gate 1 — недавняя активность в whale_trades

Трейдер должен иметь сделки за последние 24 часа.

Бизнесовый смысл:

система копирует только актуально активных трейдеров, а не исторические адреса, которые когда-то были активны.

### Gate 2 — активность за 7 дней

У трейдера должно быть значение trades_last_7_days > 0.

Бизнесовый смысл:

исключаются адреса, не имеющие подтверждённой недавней активности в qualification-модели.

### Gate 3 — принадлежность к top-50

Трейдер должен входить в топ-50 по total_volume_usd.

Бизнесовый смысл:

в paper pipeline попадает не любой qualified whale, а только ограниченный верхний слой ranked traders.

---

## 7.3 Дедупликация на уровне trigger

Перед созданием paper_trade система проверяет, не была ли похожая сделка уже создана недавно.

Условие дедупликации:

- тот же whale_address
- тот же market_id
- тот же side
- created_at в пределах последних 5 минут

Если такая запись уже есть в paper_trades, новая запись не создаётся.

Бизнесовый смысл:

система защищается от повторного копирования серии почти одинаковых сигналов по одной и той же стороне рынка.

---

## 7.4 Обработка изменения стороны сделки

Если кит меняет сторону сделки, например:

- сначала BUY
- затем SELL

то trigger рассматривает это как новый независимый сигнал.

Поскольку дедупликация учитывает side, противоположная сторона не блокируется.

Следствие:

для нового направления может быть создан новый paper_trade.

---

## 7.5 Расчёт размера paper-сделки

Система использует фиксированную sizing-логику на основе виртуального bankroll.

Используемые параметры:

- virtual bankroll = 100 USD
- kelly_fraction = 0.25
- max_position = 2% от bankroll

Промежуточный расчёт:

- kelly_size = bankroll × 0.25 = 25 USD
- max_position = bankroll × 0.02 = 2 USD

Итоговый размер сделки:

min(kelly_size, max_position) = 2 USD

Таким образом фактический размер каждой создаваемой paper-сделки ограничен 2 USD.

Бизнесовый смысл:

система использует очень консервативный position cap, который полностью ограничивает nominal Kelly sizing.

---

## 7.6 Расчёт количества контрактов

В paper_trades записываются одновременно:

- size_usd
- size

Логика:

size = size_usd / price

То есть размер позиции в контрактах рассчитывается из долларового объёма и цены входа.

---

## 7.7 Создание записи в paper_trades

Если все фильтры пройдены и дедупликация не блокирует сигнал, trigger выполняет INSERT в paper_trades.

В запись передаются:

- whale_address
- market_id
- market_title
- side
- price
- size
- size_usd
- kelly_fraction
- kelly_size
- created_at
- source

Время создания берётся из traded_at входящей сделки.

Таким образом paper_trades отражает не факт исполнения сделки, а факт допуска сигнала к paper execution.

---

## 8. Структура таблицы paper_trades

Таблица paper_trades является staging-таблицей однонаправленного потока.

Основные поля:

id — integer — первичный ключ

whale_address — text — адрес трейдера, сигнал которого копируется

market_id — text — идентификатор рынка

side — text — направление сделки

price — numeric — цена сделки

size — numeric — размер позиции в контрактах

size_usd — numeric — размер позиции в долларах

kelly_fraction — numeric — используемая доля Kelly

kelly_size — numeric — рассчитанный объём по Kelly / capped size

created_at — timestamp — время создания сигнала

source — varchar(20) — источник сигнала

market_title — text — название рынка

Индексы:

- primary key по id
- индекс по whale_address
- индекс по market_id
- индекс по created_at DESC
- dedup индекс по whale_address, market_id, side, created_at DESC

---

## 9. Статусы таблицы paper_trades

В таблице paper_trades отсутствует поле status.

Это означает, что paper_trades не хранит lifecycle сигнала.

Бизнесовый смысл таблицы:

paper_trades — это staging buffer, а не состояние сделки.

После этого execution engine принимает запись и создаёт фактическую запись в trades с exchange = VIRTUAL.

---

## 10. Использование результата downstream

Таблица paper_trades читается downstream-компонентом в src/main.py.

Модель использования:

- main.py опрашивает paper_trades каждые 5 секунд
- выполняется выборка pending candidate trades
- дополнительно используется дедупликация через таблицу trades
- если аналогичная virtual trade уже существует, повторное исполнение не происходит
- если нет, выполняется virtual execution

Таким образом paper_trades является буфером между trigger-сигналом и фактическим исполнением.

---

## 11. Особые случаи

## 11.1 Несколько одинаковых сделок подряд

Если кит делает несколько похожих сделок подряд по одному рынку и одной стороне, система пытается не создавать дубликаты в пределах 5 минут.

Это снижает шум, но не гарантирует идеальную защиту вне этого окна.

---

## 11.2 Противоположная сделка

Если кит совершает сделку противоположной стороны, trigger не блокирует её как дубликат.

Следовательно система может сформировать новый paper_trade для opposite side.

---

## 11.3 Рынок близок к резолюции

В текущей логике отсутствует фильтр по близости рынка к резолюции.

Это означает, что paper_trade может быть создан даже для рынка, который уже близок к завершению.

Это следует документировать как архитектурный риск.

---

## 11.4 Жёстко зафиксированный sizing

Несмотря на наличие Kelly-параметров, фактический размер сделки всегда ограничивается 2 USD при текущем bankroll = 100 USD.

Следствие:

kelly_fraction в текущей конфигурации практически не влияет на итоговую позицию, так как полностью доминирует cap 2%.

---

## 11.5 Отсутствие статуса у paper_trades

Так как в paper_trades нет статуса, состояние сигнала нельзя отслеживать в самой таблице.

Это усложняет аудит прохождения сигнала через execution layer и делает таблицу чисто транзитной.

---

## 12. Результат шага

Результатом шага является создание записи в paper_trades для сделки кита, прошедшей все copy filters.

Такая запись означает:

система признала сделку пригодной для paper-copy execution.

Это ещё не исполненная сделка, а только допущенный к исполнению сигнал.

- проверяет, не исполнена ли аналогичная сделка ранее
- создаёт virtual trade
- пишет запись в trades с exchange = VIRTUAL

# ШАГ 6 — EXECUTION ENGINE: ОБРАБОТКА paper_trades И ЗАПИСЬ В trades

## 1. Назначение шага

Шаг отвечает за исполнение paper-сигналов и создание фактических записей в таблице trades.

На этом шаге система:

1. периодически читает новые записи из paper_trades
2. отбирает только ещё не исполненные сигналы
3. выполняет дополнительные runtime-проверки
4. передаёт сигнал в virtual execution engine
5. создаёт запись в trades с exchange = VIRTUAL
6. открывает виртуальную позицию со status = open

Результатом шага является фактически исполненная виртуальная сделка, записанная в таблицу trades.

---

## 2. Место шага в pipeline

Общая бизнес-цепочка:

→ whale_trades  
→ qualification  
→ whales  
→ copy trigger  
→ paper_trades  
→ execution engine  
→ trades  
→ settlement  
→ bankroll

Этот шаг переводит сигнал из staging-таблицы paper_trades в реальную запись виртуальной сделки в trades.

---

## 3. Контейнер выполнения

Execution выполняется в сервисе:

polymarket_bot

Основной runtime-файл:

src/main.py

Основной исполнитель сделки:

src/strategy/virtual_bankroll.py

Таким образом шаг разделён на две части:

- orchestration и polling в main.py
- фактическое виртуальное исполнение и запись в БД в VirtualBankroll

## 4. Исходные файлы и компоненты

Основные компоненты шага:

основной execution loop, polling paper_trades, runtime checks, orchestration

src/strategy/virtual_bankroll.py  
виртуальное исполнение сделки, проверка капитала, запись в trades

Ключевые функции:

## 5. Интервал выполнения

Execution engine работает в постоянном цикле.

Модель выполнения:

- бесконечный async loop
- базовая итерация каждые 1 секунду
- чтение paper_trades каждые 5 итераций

Итоговый polling interval:

примерно каждые 5 секунд

Дополнительно используется batch limit на число обрабатываемых сигналов.

## 6. Источник данных

Непосредственный источник исполнения:

таблица paper_trades

- paper_trade_id
- market_id

- size_usd

- whale_address

Система использует только свежие записи, попавшие в окно последних 15 минут.

## 7. Алгоритм шага

## 7.1 Чтение pending paper_trades

Execution engine читает записи из paper_trades через функцию _get_pending_paper_trades().

Условия выборки:

- created_at > NOW() - INTERVAL '15 minutes'
- отсутствует уже исполненная аналогичная сделка в trades
- сортировка по created_at DESC
- limit = 10

Бизнесовый смысл:

система обрабатывает только свежие сигналы, которые ещё не были исполнены как virtual trades.

---

## 7.2 SQL-дедупликация при выборке

Уже на этапе чтения paper_trades выполняется фильтр через NOT EXISTS against trades.

Сделка не попадает в исполнение, если в trades уже существует запись, у которой совпадают:

- market_id
- whale_source
- side
- exchange = VIRTUAL

Это первый execution-level механизм защиты от повторного исполнения.

---

## 7.3 Runtime duplicate check

После чтения сигнала выполняется дополнительная runtime-проверка через _check_trade_exists().

Эта проверка использует более узкий matching по:

- market_id
- whale_source
- size
- price
- exchange = VIRTUAL

Бизнесовый смысл:

система повторно убеждается, что такая virtual trade ещё не была сохранена.

---

## 7.4 In-memory tracking

Дополнительно execution loop использует in-memory set processed trade ids.

Это ещё один защитный слой на уровне процесса.

Следствие:

в рамках жизни конкретного процесса уменьшается риск повторной обработки одного и того же сигнала.

---

## 7.5 Проверка zero-size

Перед исполнением система выполняет проверку размера сделки.

Если size_usd <= 0, сделка пропускается.

Бизнесовый смысл:

сигналы с нулевым или некорректным размером не исполняются.

Это важно, потому что ранее в проекте уже наблюдалась проблема zero-size trades.

---

## 7.6 Передача сигнала в VirtualBankroll

После прохождения всех проверок сигнал передаётся в VirtualBankroll.execute_virtual_trade().

На этом этапе execution engine:

- интерпретирует сигнал как виртуальную сделку
- проверяет возможность открытия или закрытия позиции
- рассчитывает параметры исполнения
- сохраняет сделку в таблицу trades

---

## 7.7 Проверка капитала и корректности сделки

Внутри VirtualBankroll выполняются валидации:

- sufficient capital
- корректность side
- существование позиции для sell
- корректность параметров исполнения

Если капитала недостаточно, выбрасывается ошибка.

Если side невалидный, выбрасывается ошибка.

Если sell выполняется без позиции, выбрасывается ошибка.

---

## 7.8 Создание записи в trades

После успешного виртуального исполнения вызывается _save_virtual_trade().

В trades записываются:

- trade_id
- market_id
- side
- size
- price
- exchange
- commission
- gas_cost_eth
- gas_cost_usd
- net_pnl
- status
- executed_at
- settled_at
- opportunity_id
- fiat_fees
- gross_pnl
- total_fees
- market_title
- whale_source

Связь с источником сигнала сохраняется через поле:

whale_source = whale_address

Это критически важно для последующего анализа стратегии и deduplication.

---

## 7.9 Начальный статус сделки

При создании виртуальной сделки система записывает:

- exchange = VIRTUAL
- status = open

Если сделка создаётся как открывающая позицию, она получает status open.

Закрытие позиции и перевод в closed происходят уже на последующих этапах settlement pipeline.

---

## 7.10 UPSERT по trade_id

Запись в trades выполняется с ON CONFLICT (trade_id) DO UPDATE.

Это означает, что trade_id является DB-level ключом для самой записи trades.

Бизнесовый смысл:

если одна и та же trade_id повторно сохраняется, система выполняет обновление вместо второго INSERT.

## 8. Структура таблицы trades

Таблица trades хранит фактически исполненные virtual или live сделки.

## 9. Механизмы дедупликации

Execution step использует четыре уровня защиты от дубликатов.

### Level 1 — trigger-level deduplication

Ещё до попадания в execution слой paper_trades защищён от части дубликатов на шаге copy trigger через 5-минутное окно.

### Level 2 — SQL NOT EXISTS against trades

При чтении paper_trades система исключает сигналы, уже имеющие соответствующую virtual trade.

### Level 3 — runtime duplicate check

Перед исполнением выполняется дополнительная проверка существования trades по market_id, whale_source, size и price.

### Level 4 — in-memory processed set

В процессе выполнения используется in-memory tracking уже обработанных trade ids.

Бизнесовый смысл:

защита от повторного исполнения реализована многоуровнево и распределена между шагами pipeline.

---

## 10. Особые случаи

## 10.1 Zero-size trade

Если размер сделки меньше либо равен нулю, execution engine пропускает сигнал.

Это корректная защитная логика.

---

## 10.2 Price = 0

В текущем execution flow нет явной проверки price = 0 до исполнения.

Это риск, потому что price участвует в расчётах позиции и стоимости входа.

Эту проблему нужно отдельно зафиксировать как архитектурный риск.

---

## 10.3 Closed market

В текущем execution flow нет явной проверки, закрыт ли рынок на момент исполнения.

Следствие:

виртуальная сделка может быть создана по сигналу рынка, который уже фактически не должен открываться.

Это также следует фиксировать как риск.

---

## 10.4 Duplicate trade

Если система обнаруживает, что аналогичная virtual trade уже существует, сигнал пропускается.

Это реализовано и на SQL уровне, и на runtime уровне.

---

## 10.5 Insufficient capital

Если свободного виртуального капитала недостаточно, execution engine выбрасывает ошибку и сделка не исполняется.

---

## 10.6 Sell without position

Если приходит сигнал sell без существующей позиции, VirtualBankroll отклоняет исполнение.

---

## 11. Результат шага

Результатом шага является запись в таблицу trades, представляющая фактически исполненную виртуальную сделку.

Ключевые атрибуты результата:

- exchange = VIRTUAL
- status = open
- whale_source сохранён
- executed_at заполнен

Бизнесово это означает:

сигнал из paper_trades стал активной виртуальной позицией системы.

---

## 12. Использование результата

Следующий шаг pipeline:

ШАГ 7 — settlement / закрытие позиции и перевод trades из open в closed

На этом шаге система:

- отслеживает резолюцию рынка
- определяет финальный исход
- закрывает позицию
- рассчитывает PnL
- обновляет bankroll

paper_trades
